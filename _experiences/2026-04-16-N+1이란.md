# N+1이란

- **N+1 문제**는 **JPA**나 **Hibernate**와 같은 ORM을 사용할 때 발생하는 성능 문제로, 연관된 엔티티를 조회할 때 **의도하지 않게 다수의 쿼리가 발생하는 현상**을 의미.
- 특히, **일대다** 또는 **다대일** 관계에서 자주 발생하며, 기본 엔티티를 조회한 후, 연관된 엔티티를 각각 조회하기 위해 추가적인 쿼리가 발생하는 상황을 말함.
- **해결 방법**으로는 페치 조인(`JOIN FETCH`), `@EntityGraph`, `@BatchSize`, 그리고 지연 로딩(`FetchType.LAZY`)을 사용하는 방법이 있음.
- **페치 조인**은 가장 일반적인 해결책으로, 한 번의 쿼리로 연관된 엔티티를 함께 조회하여 성능을 최적화.

---

# **N+1 문제 설명**

- 예를 들어, **N개의 부모 엔티티**가 있을 때, 각 부모 엔티티에 연관된 **자식 엔티티를 조회**해야 하는 상황이 있다고 가정.
- 먼저 **부모 엔티티**를 조회하는 **1개의 쿼리**가 실행됨.
- 그 후, 각 부모 엔티티와 연관된 **자식 엔티티**를 조회하기 위해 **N개의 추가 쿼리**가 발생함.
- 결과적으로 **총 N+1개의 쿼리**가 실행됨.

```jsx
// 부모 엔티티 조회
List<Parent> parents = em.createQuery("SELECT p FROM Parent p", Parent.class).getResultList();

// 부모 엔티티의 자식 엔티티 접근
for (Parent parent : parents) {
    parent.getChildren().size();  // 자식 엔티티 조회 (추가 쿼리 발생)
}

```

- 위 코드에서 첫 번째 쿼리로 `Parent` 엔티티들을 가져온 후, 각 `Parent` 엔티티의 `Children` 필드에 접근할 때마다 별도의 쿼리가 발생하여 **N+1 문제**가 발생하게 됨.

---

# **페치 조인(Fetch Join) 사용**

- **JPQL**에서 **페치 조인**을 사용하여 한 번의 쿼리로 연관된 엔티티를 함께 조회할 수 있음. 이를 통해 **N+1 문제**를 해결할 수 있음.
- **설명**: 페치 조인은 연관된 엔티티를 **즉시 로딩**하는 방식으로, 한 번의 쿼리로 부모와 자식 엔티티를 함께 조회.
- **장점**: 추가적인 쿼리가 발생하지 않으며 성능을 최적화할 수 있음.
- JPQL
    
    ```java
    // JPQL을 사용한 페치 조인
    List<Parent> parents = em.createQuery(
        "SELECT p FROM Parent p JOIN FETCH p.children", Parent.class)
        .getResultList();
    
    ```
    
- QueryDSL
    
    ```java
    @Repository
    public class ParentRepositoryCustomImpl implements ParentRepositoryCustom {
    
        private final JPAQueryFactory queryFactory;
    
        public ParentRepositoryCustomImpl(EntityManager em) {
            this.queryFactory = new JPAQueryFactory(em);
        }
    
        public List<Parent> findAllWithChildren() {
            QParent parent = QParent.parent;
            QChild child = QChild.child;
    
            return queryFactory
                .selectFrom(parent)
                .leftJoin(parent.children, child).fetchJoin()  // 페치 조인
                .fetch();  // 결과 조회
        }
    }
    
    ```
    

위 쿼리는 **부모 엔티티와 자식 엔티티**를 한 번에 조회하므로 추가 쿼리가 발생하지 않음.

# `@EntityGraph` 사용

- `@EntityGraph`를 사용하여 페치 전략을 동적으로 설정할 수 있음.
- JPQL을 사용하지 않고도 지연 로딩(LAZY으로 설정된 연관 엔티티를 즉시 로딩할 수 있는 방법.
- **설명**: 엔티티 그래프는 특정 엔티티를 조회할 때 어떤 연관된 엔티티를 즉시 로딩할지 지정할 수 있음.
- **장점**: JPQL 사용 없이도 코드에서 필요한 경우에만 페치 전략을 동적으로 변경 가능.
- 코드
    
    ```java
    @EntityGraph(attributePaths = {"children"})
    List<Parent> parents = parentRepository.findAll();
    
    ```
    

위 코드에서 `Parent` 엔티티를 조회할 때, `children` 속성도 함께 페치 조인하여 가져옴.

# `@BatchSize` 사용

- `@BatchSize`는 **N+1 문제**를 완화하는 데 도움이 되는 방법으로, **배치로 조회**하여 여러 개의 연관 엔티티를 한 번에 가져옴. 지연 로딩일 때 쿼리의 개수를 줄이는 방식.
- **설명**: `@BatchSize`는 한 번에 여러 엔티티를 가져오는 쿼리를 배치로 실행하여 쿼리 수를 줄이는 방법.
- **장점**: 지연 로딩을 유지하면서도 성능 최적화 가능.
- 코드
    
    ```java
    @Entity
    public class Parent {
    
        @OneToMany(mappedBy = "parent")
        @BatchSize(size = 10)
        private List<Child> children;
    }
    
    ```
    

위 설정으로 한 번에 10개의 자식 엔티티를 조회하게 되어 쿼리 수를 줄일 수 있음.

# `@ManyToOne`에서 `FetchType.LAZY` 사용

**다대일(N:1)** 관계에서는 기본적으로 `FetchType.EAGER`로 설정되는데, 이를 **`FetchType.LAZY`**로 설정하여 실제로 필요할 때만 연관 엔티티를 조회할 수 있음.

- **설명**: 즉시 로딩(EAGER)을 사용하면 N+1 문제가 발생할 수 있으므로, 지연 로딩(LAZY)로 설정하여 필요한 시점에만 연관된 엔티티를 로드.
- **장점**: 필요할 때만 쿼리를 실행하여 성능을 최적화.
- 코드
    
    ```java
    @ManyToOne(fetch = FetchType.LAZY)
    private Parent parent;
    
    ```
