# API에서의 에러 핸들링을 어떻게 구현하는가

- API에서 에러 핸들링은 클라이언트가 요청 실패의 원인을 정확히 이해하고 적절히 대응할 수 있도록 명확한 정보를 제공하는 과정.
- 잘 설계된 에러 핸들링은 API의 사용자 경험과 디버깅 효율성을 크게 향상시킴.

---

# API 에러 핸들링의 기본 원칙

## **일관성**

- 에러 메시지의 구조와 형식을 일관되게 유지해야 함.
- 모든 에러에 동일한 포맷을 적용해 클라이언트가 예측 가능하도록 설계.

## **명확성**

- 에러 원인을 명확히 기술하여 클라이언트가 쉽게 이해할 수 있도록 함.
- 불필요한 기술적 용어를 줄이고, 에러 메시지를 간결하게 작성.

## **HTTP 상태 코드 활용**

- 각 요청 결과에 적절한 HTTP 상태 코드를 반환.

## **보안**

- 에러 메시지에 민감한 정보를 포함하지 않도록 주의.

---

# HTTP 상태 코드를 활용한 에러 핸들링

## **2xx (성공)**

- `200 OK`: 요청이 성공적으로 처리됨.
- `201 Created`: 새로운 리소스가 생성됨.

## **4xx (클라이언트 에러)**

- `400 Bad Request`: 잘못된 요청.
- `401 Unauthorized`: 인증이 필요함.
- `403 Forbidden`: 권한 부족.
- `404 Not Found`: 리소스가 존재하지 않음.
- `409 Conflict`: 리소스 충돌 발생.
- `422 Unprocessable Entity`: 요청 데이터가 처리 불가능.

## **5xx (서버 에러)**

- `500 Internal Server Error`: 서버에서 예기치 못한 에러 발생.
- `502 Bad Gateway`: 게이트웨이 또는 프록시 서버 에러.
- `503 Service Unavailable`: 서버가 현재 요청을 처리할 수 없음.

# 에러 응답의 구조 설계

## **예시: JSON 기반 에러 응답**

```json
{
  "status": 400,
  "error": "Bad Request",
  "message": "The 'email' field is required.",
  "details": {
    "field": "email",
    "issue": "missing"
  },
  "timestamp": "2024-11-30T12:34:56Z"
}
```

- **status**: HTTP 상태 코드.
- **error**: 에러 유형 또는 이름.
- **message**: 클라이언트에 표시할 간단한 에러 설명.
- **details**: 추가 정보(필드 이름, 발생 이유 등).
- **timestamp**: 에러 발생 시간.

# 구현 예시: Express를 이용한 에러 핸들링

## 1. **기본적인 에러 핸들링**

```jsx
const express = require('express');
const app = express();

// 라우트 예시
app.get('/users/:id', (req, res, next) => {
    const userId = req.params.id;
    if (!userId) {
        // 에러 발생
        next({ status: 400, message: "User ID is required" });
    }
    // 정상 응답 처리
    res.json({ id: userId, name: "John Doe" });
});

// 에러 처리 미들웨어
app.use((err, req, res, next) => {
    res.status(err.status || 500).json({
        status: err.status || 500,
        error: err.error || "Internal Server Error",
        message: err.message || "Something went wrong",
        timestamp: new Date().toISOString()
    });
});

app.listen(3000, () => console.log("Server is running"));
```

## 2. **유효성 검사와 상세 에러 제공**

```jsx
const Joi = require('joi');

app.post('/users', (req, res, next) => {
    const schema = Joi.object({
        name: Joi.string().required(),
        email: Joi.string().email().required()
    });

    const { error } = schema.validate(req.body);
    if (error) {
        return next({
            status: 422,
            message: "Validation failed",
            details: error.details.map(detail => detail.message)
        });
    }

    res.status(201).json({ message: "User created successfully" });
});
```

# 추가적인 에러 핸들링 전략

## **로깅(Log)**

- 서버에서 발생한 에러를 로깅 시스템(Sentry, ELK 등)에 기록하여 디버깅과 분석에 활용.

## **에러 코드 정의**

- API 고유의 에러 코드를 정의해 클라이언트가 에러 원인을 더 쉽게 파악하도록 지원.

```json
{
  "status": 404,
  "error_code": "USER_NOT_FOUND",
  "message": "The requested user does not exist."
}
```

## **Graceful Fallback**

- 서버 에러 발생 시 기본적인 기능이라도 작동하도록 처리.
- 예: 검색 결과가 없을 때 기본 메시지를 반환.

## **Retry 및 Backoff**

- 서버 과부하 시 클라이언트가 자동으로 재시도하도록 지연(Backoff)을 구현.

---

# 장점

- 클라이언트는 명확한 에러 정보를 통해 문제를 이해하고 대응 가능.
- 디버깅과 모니터링이 용이해져 시스템 안정성이 향상.
- 일관된 구조로 유지보수와 확장성 증대.

# 단점

- 세부적인 에러 처리를 구현하려면 추가적인 노력과 시간이 필요.
- 민감한 정보를 노출하지 않도록 세심한 관리가 필요.

---

# **요약**

- API에서의 에러 핸들링은 HTTP 상태 코드, 명확한 에러 메시지, 구조화된 응답을 활용해 클라이언트가 요청 실패 원인을 이해하도록 돕는 것이 핵심.
- 서버 에러는 로그와 모니터링으로 관리하며, 일관성 있는 설계와 보안을 고려해야 함.
