# 배치(Batch) vs 큐(Queue)

## 개요

데이터 처리 방식은 크게 "언제 처리하느냐"와 "어떻게 처리하느냐"로 나뉩니다. 배치와 큐는 그 두 축을 대표하는 패턴으로, 각각의 철학과 트레이드오프가 명확합니다. 최근 큐 기반 처리가 늘어나는 이유는 단순히 유행이 아니라, 시스템 복잡도와 운영 요구사항의 변화에서 비롯됩니다.

---

## 배치(Batch) 처리

### 개념

배치는 "데이터를 모아서 한꺼번에 처리"하는 방식입니다. 처리할 대상을 일정 기준(시간, 건수, 크기)으로 묶어 한 번에 실행합니다.

### 특징

#### 실행 모델

- 스케줄 기반으로 동작합니다 (cron, Jenkins 등)
- 시작과 끝이 명확한 "Job" 단위입니다
- 실행 중에는 입력 데이터가 고정됩니다

#### 장점

- 대용량 데이터를 효율적으로 처리할 수 있습니다 (I/O 최적화, 벌크 쿼리 등)
- 처리 흐름이 단순하고 디버깅이 쉽습니다
- 리소스 사용 시점을 예측하고 제어하기 쉽습니다 (야간 실행 등)

#### 단점

- 처리 지연이 발생합니다 — 데이터가 생성되더라도 다음 배치 시점까지 기다려야 합니다
- 장애 시 재처리 범위가 넓어질 수 있습니다
- 부분 실패 처리(partial failure)가 까다롭습니다

### 대표 사용 사례

- 야간 정산, 월말 청구서 생성
- ETL (데이터 웨어하우스 적재)
- 대용량 리포트 생성

## 큐(Queue) 기반 처리

### 개념

큐는 "이벤트 또는 메시지가 발생하는 즉시 처리 파이프라인에 넣는" 방식입니다. 생산자(Producer)와 소비자(Consumer)가 분리되어 비동기로 동작합니다.

### 특징

#### 실행 모델

- 이벤트 발생 즉시 큐에 적재됩니다
- Consumer가 메시지를 꺼내 처리합니다
- 생산과 소비가 독립적인 속도로 동작합니다

#### 장점

- 처리 지연이 낮습니다 — 이벤트 발생 직후 처리 가능합니다
- Consumer를 수평 확장하기 쉽습니다
- Producer/Consumer 간 결합도가 낮아 유연합니다
- 실패한 메시지만 재처리하면 됩니다 (DLQ, Dead Letter Queue)
- 시스템 간 버퍼 역할을 해서 트래픽 스파이크를 흡수합니다

#### 단점

- 인프라가 추가됩니다 (Kafka, RabbitMQ, SQS 등)
- 메시지 순서 보장, 중복 처리(exactly-once) 등 고려할 복잡도가 생깁니다
- 메시지 유실/적체 모니터링이 필요합니다

### 대표 사용 사례

- 주문 처리, 알림 발송
- 마이크로서비스 간 이벤트 전파
- 실시간 로그 수집 및 처리

---

## 비교 요약

| 항목 | 배치 | 큐 |
| --- | --- | --- |
| 처리 시점 | 주기적 (스케줄) | 즉시 (이벤트 발생 시) |
| 처리 단위 | 묶음(Job) | 개별 메시지 |
| 지연(Latency) | 높음 | 낮음 |
| 처리량(Throughput) | 높음 (벌크 최적화) | 상황에 따라 다름 |
| 확장성 | 수직/수평 모두 가능 | 수평 확장에 유리 |
| 재처리 단위 | Job 전체 or 체크포인트 | 개별 메시지 (DLQ) |
| 운영 복잡도 | 낮음 | 중간~높음 |
| 인프라 의존성 | 낮음 | 메시지 브로커 필요 |
| 장애 영향 범위 | 넓음 | 좁음 |

## 왜 요즘 큐가 더 많이 쓰이는가

### 운영 관리의 편의성 변화

예전에는 배치가 "관리가 쉬운" 선택이었습니다. 실행 로직이 단일 프로세스에 집중되어 있고, cron 하나면 충분했습니다.

지금은 반대입니다. Kafka, SQS, RabbitMQ 같은 managed 서비스가 보편화되면서 인프라 관리 부담이 크게 줄었습니다. 반면 배치는 "언제 실패했는지", "어디서 멈췄는지"를 추적하는 것이 점점 복잡해졌습니다.

### 아키텍처 트렌드와의 정합성

마이크로서비스 환경에서는 서비스 간 직접 호출 대신 이벤트 기반 통신이 자연스럽습니다. 큐는 이 구조에 그대로 녹아들지만, 배치는 서비스 경계를 넘나들수록 조율이 어렵습니다.

### 사용자 기대치의 변화

사용자들은 "다음날 아침에 처리됩니다"보다 "즉시 처리됩니다"를 기대합니다. 실시간성에 대한 요구가 높아지면서, 배치가 적합한 도메인이 점점 좁아지고 있습니다.

---

## 전체 요약

배치는 "모아서 한 번에", 큐는 "발생하면 즉시"라는 근본적인 철학의 차이를 가집니다. 배치는 단순하고 대용량에 강하지만 지연과 재처리 범위가 약점입니다. 큐는 낮은 지연, 유연한 확장, 세밀한 재처리가 강점이지만 메시지 브로커라는 인프라와 그에 따른 운영 고려사항이 생깁니다.

요즘 큐가 선호되는 핵심 이유는 "managed 서비스 보편화로 인프라 부담 감소" + "마이크로서비스 아키텍처와의 자연스러운 정합성" + "실시간성에 대한 사용자 기대 증가" 세 가지로 정리할 수 있습니다. 그럼에도 대용량 데이터 처리, 야간 정산처럼 배치가 본질적으로 더 적합한 도메인은 여전히 존재합니다.
