## 개요

S3나 GCS 같은 오브젝트 스토리지에 이미지를 올리면 기본적으로 "https://my-bucket.s3.ap-northeast-2.amazonaws.com/uploads/abc.jpg" 같은 버킷 고유의 URL이 노출됩니다. 반면 많은 서비스들은 "image.domain.com/file/abc.jpg" 같은 형태로 자사 도메인을 노출시키는데, 이는 별도의 이미지 API를 새로 만든 것이 아니라 CDN을 버킷 앞에 세우고 커스텀 도메인을 연결한 결과입니다.

---

## CDN을 통한 도메인 매스킹

### CDN의 역할

CDN (Content Delivery Network)은 원본 서버 (Origin, 여기서는 S3 버킷) 앞단에 위치하는 캐시 및 라우팅 계층입니다. 사용자의 요청은 버킷에 직접 가지 않고 CDN을 거치게 되며, CDN이 버킷에서 파일을 가져와 캐싱한 후 사용자에게 전달합니다.

#### 동작 흐름

요청 흐름은 다음과 같이 구성됩니다.

```
사용자 → image.domain.com (CDN) → my-bucket.s3.amazonaws.com (Origin)
```

사용자 입장에서는 "image.domain.com"만 보이고, 실제 버킷 주소는 CDN 뒤에 숨겨진 채로 응답이 캐싱되어 빠르게 내려갑니다.

#### 대표적인 CDN 서비스

이 구조를 구현하는 데 흔히 쓰이는 서비스로는 AWS CloudFront, Cloudflare, Fastly, Akamai 등이 있습니다. AWS 환경이라면 CloudFront가 S3와 가장 자연스럽게 연동되고, 멀티 클라우드나 비용 효율을 고려한다면 Cloudflare가 많이 선택됩니다.

### 커스텀 도메인 연결 방법

#### CloudFront 기준 설정 절차

CloudFront를 예로 들면 다음과 같은 순서로 설정합니다.

1. CloudFront Distribution을 생성하고, Origin으로 S3 버킷을 지정합니다.
2. ACM (AWS Certificate Manager)에서 "image.domain.com"에 대한 SSL 인증서를 발급받습니다. (CloudFront에 연결할 인증서는 반드시 us-east-1 리전에서 발급해야 합니다.)
3. CloudFront의 "Alternate Domain Name (CNAME)" 설정에 "image.domain.com"을 등록하고, 위에서 발급한 인증서를 연결합니다.
4. DNS에서 "image.domain.com"을 CloudFront가 제공하는 도메인 ("xxxx.cloudfront.net")으로 CNAME 레코드를 설정합니다.

이렇게 하면 사용자는 "image.domain.com/uploads/abc.jpg" 형태로 접근하게 되고, 실제로는 CloudFront가 뒤에서 S3 버킷의 파일을 가져와 응답합니다.

#### S3 버킷 접근 제한

CDN을 거치도록 강제하려면, S3 버킷에 직접 접근하는 것을 막아야 합니다. 이를 위해 OAC (Origin Access Control, 구버전은 OAI)를 사용해서 "CloudFront를 통해서만 버킷에 접근 가능"하도록 버킷 정책을 설정하는 것이 일반적입니다. 이렇게 하지 않으면 원본 S3 URL로도 여전히 접근이 가능해서, 도메인을 숨긴 의미가 줄어듭니다.

## 단순 매스킹과 이미지 처리 API의 차이

### 단순 리버스 프록시 방식

위에서 설명한 CDN 연결만으로도 도메인을 숨기는 목적은 달성됩니다. 이 경우 "image.domain.com/file/abc.jpg"로 접근하면 원본 파일이 그대로 전달될 뿐, 별도의 가공은 일어나지 않습니다.

### 이미지 처리 API가 추가된 경우

"다른 사이트들"이 보여주는 것이 단순히 도메인만 가려진 것이 아니라, URL 파라미터로 리사이징이나 포맷 변환까지 지원하는 경우 (예: "image.domain.com/file/abc.jpg?w=300&h=300&format=webp")라면라면) 이는 CDN 뒤에 이미지 처리 레이어가 추가로 붙어있는 구조입니다.

#### Lambda@Edge / CloudFront Functions

AWS 환경에서는 CloudFront의 요청/응답 단계에 Lambda@Edge나 CloudFront Functions를 붙여서, 요청 URL의 쿼리 파라미터를 읽고 원본 이미지를 동적으로 리사이징한 뒤 캐싱하는 패턴이 흔히 쓰입니다.

#### 전용 이미지 최적화 서비스

직접 구축하지 않고 관리형 서비스를 쓰는 경우도 많습니다. 대표적으로 Cloudflare Images, Imgix, Cloudinary 같은 서비스들이 있으며, 이들은 버킷이나 원본 스토리지를 자신들의 CDN 뒤에 연결하고, URL 파라미터 기반의 이미지 변환 API를 함께 제공합니다.

#### 두 방식의 비교

| 항목 | 단순 CDN 매스킹 | 이미지 처리 API 포함 |
| --- | --- | --- |
| 목적 | 도메인 숨김, 캐싱 | 도메인 숨김 + 동적 변환 |
| 구현 난이도 | 낮음 | 중간 ~ 높음 |
| 대표 도구 | CloudFront, Cloudflare CDN | Lambda@Edge, Cloudinary, Imgix |
| 추가 비용 | CDN 트래픽 비용 | CDN 비용 + 처리 비용 |

## Spring Boot 환경에서의 적용 방향

현재 사용 중인 스택을 고려하면, 이미지 업로드 후 반환하는 URL을 버킷 URL이 아니라 CDN 도메인으로 구성하는 방식이 적합합니다.

### 적용 방식

업로드 로직에서 S3에 파일을 저장한 뒤, 클라이언트에게 내려주는 URL은 버킷 URL이 아니라 "https://image.domain.com/{key}" 형태로 조합해서 반환하는 것이 핵심입니다. 즉 저장 위치 (S3)와 응답 URL (CDN 도메인)을 분리해서 관리하는 구조입니다.

```kotlin
@Value("\${app.cdn.base-url}")
lateinit var cdnBaseUrl: String

fun buildPublicUrl(s3Key: String): String {
    return "$cdnBaseUrl/$s3Key"
}
```

이렇게 하면 추후 CDN 업체를 변경하거나 버킷을 이전해도, "app.cdn.base-url" 설정값만 바꾸면 되고 클라이언트에 노출되는 URL 패턴은 그대로 유지할 수 있습니다.

## 전체 요약

버킷 주소가 그대로 노출되는 문제는 CDN을 버킷 앞에 두고 커스텀 도메인을 연결하는 방식으로 해결합니다. AWS 환경에서는 CloudFront + ACM 인증서 + OAC 조합이 일반적이며, S3에 대한 직접 접근은 막고 CDN을 통해서만 접근하도록 구성합니다. 만약 URL 파라미터로 리사이징이나 포맷 변환까지 지원하는 사이트라면, 이는 CDN 뒤에 Lambda@Edge나 Cloudinary 같은 이미지 처리 레이어가 추가로 붙은 구조입니다. 즉 핵심은 "별도의 이미지 API를 새로 만든 것"이 아니라, "CDN으로 도메인을 가리고, 필요하면 그 위에 변환 기능을 얹은 것"입니다.
