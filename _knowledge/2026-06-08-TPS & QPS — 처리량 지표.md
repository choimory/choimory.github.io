# TPS / QPS — 처리량 지표 개념 완전 정리

## 개요

TPS와 QPS는 분산 시스템과 백엔드 서비스를 설계할 때 가장 기본이 되는 성능 지표입니다. 이 두 지표는 시스템이 단위 시간당 얼마나 많은 작업을 처리할 수 있는지를 나타내며, 용량 계획(Capacity Planning), SLA 정의, 부하 테스트의 기준이 됩니다.

---

## TPS (Transactions Per Second)

### 개념

TPS는 초당 처리되는 "트랜잭션" 수를 의미합니다. 여기서 트랜잭션은 단순한 요청 하나가 아니라, 비즈니스적으로 의미 있는 하나의 작업 단위를 말합니다.

예를 들어 "주문 생성" 하나는 다음과 같은 여러 작업을 포함합니다.

- DB INSERT (주문 테이블)
- DB UPDATE (재고 테이블)
- Kafka 이벤트 발행
- Redis 캐시 무효화

이 모든 과정이 하나의 트랜잭션으로 묶이며, TPS는 이 단위가 초당 몇 번 완료되는지를 측정합니다.

### 특성

TPS는 주로 데이터베이스, 결제 시스템, 주문 처리 시스템처럼 "상태 변경"이 수반되는 작업에서 사용합니다. 트랜잭션의 범위(scope)를 어떻게 정의하느냐에 따라 수치가 달라질 수 있기 때문에, 팀 내에서 트랜잭션 정의를 먼저 합의하는 것이 중요합니다.

---

## QPS (Queries Per Second)

### 개념

QPS는 초당 처리되는 "쿼리" 수를 의미합니다. 여기서 쿼리는 주로 조회성 요청이나 단일 처리 단위를 가리킵니다.

예를 들어 다음과 같은 것들이 QPS로 측정됩니다.

- HTTP GET 요청 수
- DB SELECT 실행 수
- Redis GET 호출 수
- Elasticsearch 검색 요청 수

### 특성

QPS는 읽기 중심의 서비스, API Gateway, 검색 서비스, CDN 등에서 많이 사용됩니다. 트랜잭션과 달리 원자성(Atomicity)이나 롤백 개념이 없는 단순 처리 단위에 적합합니다.

---

## TPS vs QPS 비교

### 개념적 차이

| 항목 | TPS | QPS |
| --- | --- | --- |
| 단위 | 비즈니스 트랜잭션 | 개별 쿼리/요청 |
| 복잡도 | 여러 작업 포함 가능 | 단일 처리 단위 |
| 주 사용처 | 결제, 주문, 상태 변경 | 조회, 검색, API 호출 |
| 상태 변경 | ✅ 포함 | 보통 ❌ |
| 원자성 보장 | ✅ 필요 | ❌ 불필요 |

### 관계

하나의 트랜잭션(TPS)은 내부적으로 여러 쿼리(QPS)를 포함할 수 있습니다.

```
1 TPS (주문 생성)
  ├── 1 QPS (재고 SELECT)
  ├── 1 QPS (주문 INSERT)
  ├── 1 QPS (재고 UPDATE)
  └── 1 QPS (이벤트 발행)
```

따라서 TPS 대비 QPS는 항상 더 크거나 같습니다.

---

## 실제 시스템에서의 수치 감각

### 일반적인 참고 수치

| 시스템 유형 | 일반적 TPS/QPS 범위 |
| --- | --- |
| 소규모 API 서버 (단일 인스턴스) | 수십 ~ 수백 TPS |
| MySQL (OLTP, 튜닝 기준) | 수백 ~ 수천 TPS |
| Redis (단일 노드) | 수만 ~ 10만 QPS |
| Kafka (단일 브로커) | 수만 ~ 수십만 QPS |
| 대형 검색 서비스 (Elasticsearch) | 수천 ~ 수만 QPS |

### Spring Boot + Kotlin 서비스 기준

일반적인 Spring Boot 서버의 TPS는 하드웨어, DB 커넥션 풀, 비즈니스 로직 복잡도에 크게 의존합니다. DB 없이 단순 응답이면 수천 TPS, DB 트랜잭션이 포함되면 수백 TPS 수준이 일반적입니다.

## 처리량에 영향을 미치는 요소

### 병목 지점 분석

TPS/QPS가 예상보다 낮을 때는 다음 순서로 병목을 탐색합니다.

```
CPU 사용률 → 메모리 → DB 커넥션 풀 → I/O → 네트워크
```

### 주요 제한 요인

- "DB 커넥션 풀 부족" : 커넥션 대기(waiting)로 스레드가 블로킹되어 전체 TPS 저하
- "GC 압박" : JVM 기반 서비스에서 Full GC 발생 시 순간적 처리량 급락
- "외부 API 응답 지연" : 동기 호출 시 스레드 점유로 처리량 감소
- "락 경합(Lock Contention)" : 동시 트랜잭션 간 DB 락 충돌 시 TPS 급감

---

## 용량 계획(Capacity Planning)에서의 활용

### DAU → TPS 역산 방법

실제 서비스 설계 시 DAU(일 활성 사용자)로부터 필요한 TPS를 역산합니다.

```
DAU: 100만 명
1인 하루 평균 요청: 50건
하루 총 요청: 5,000만 건

피크 타임 집중 비율: 하루의 20% 시간대에 80% 요청 집중
피크 초당 요청: 50,000,000 × 0.8 / (86,400 × 0.2)
              ≒ 2,315 QPS

안전 마진 (×2~3 고려): 목표 설계 처리량 ≒ 5,000~7,000 QPS
```

### 스케일링 전략

- "Scale Up" : 단일 서버 스펙 향상 → 단기 대응에 적합
- "Scale Out" : 수평 확장 (인스턴스 추가) → MSA 환경에서 선호
- "캐싱 계층 추가" : Redis 등으로 DB 부하를 QPS 기준 분산
- "비동기 처리" : Kafka 등으로 피크 시점의 TPS 버퍼링

---

## 요약

TPS와 QPS는 시스템 처리 능력을 나타내는 핵심 지표입니다. TPS는 비즈니스 트랜잭션 단위의 처리량으로 상태 변경이 수반되는 작업에, QPS는 개별 요청/쿼리 단위의 처리량으로 조회 중심 작업에 사용됩니다. 하나의 트랜잭션 안에 여러 쿼리가 포함되므로 QPS ≥ TPS 관계가 성립합니다. 실제 설계에서는 DAU로부터 필요한 TPS/QPS를 역산하고, 병목 지점을 분석하여 스케일링 전략과 캐싱 전략을 결합하는 것이 중요합니다.
