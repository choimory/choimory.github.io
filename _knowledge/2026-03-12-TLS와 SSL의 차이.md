# TLS와 SSL의 차이

- *TLS(Transport Layer Security)*와 **SSL(Secure Sockets Layer)**는 **네트워크 통신 보안 프로토콜**로, 인터넷을 통한 데이터 전송 시 **기밀성**, **무결성**, **인증**을 제공하는 기술이다.
- SSL이 먼저 개발되었고, 이후 TLS로 발전했다.
- 이 두 프로토콜은 기능적으로 유사하지만, **보안 수준**과 **프로토콜 구조**에서 차이가 있다.

---

# **SSL(Secure Sockets Layer)**

## 정의:

- **SSL**은 넷스케이프(Netscape)에 의해 개발된 **초기 보안 프로토콜**로, 클라이언트와 서버 간의 **데이터 암호화**를 통해 통신의 보안성을 보장하기 위해 만들어졌다.
- SSL은 인터넷에서 **HTTP, FTP, 이메일** 등의 애플리케이션 프로토콜을 암호화하는 데 사용되었다.

## SSL의 발전 단계:

- **SSL 1.0**:
    - 초기 버전이었지만, 보안 문제로 인해 공개되지 않았음.
- **SSL 2.0**:
    - 1995년에 발표되었으나, **보안 취약점**으로 인해 SSL 3.0으로 빠르게 대체됨.
- **SSL 3.0**:
    - 1996년에 발표되었으며, SSL 프로토콜 중 가장 널리 사용된 버전이다.
    - 그러나 시간이 지나면서 **보안상의 약점**이 발견되었고, SSL 3.0 이후에는 **TLS**로 전환되었다.

## SSL의 종료:

- **SSL 3.0**은 많은 보안 취약점이 발견되면서 **POODLE**과 같은 공격에 취약하다는 것이 밝혀졌고, SSL은 **보안 기준**으로 더 이상 사용되지 않는다.
- 대부분의 시스템에서 SSL은 **지원 중단**된 상태이다.

---

# **TLS(Transport Layer Security)**

## 정의:

- **TLS**는 SSL 3.0을 기반으로 **개선된 보안 프로토콜**로, **IETF(Internet Engineering Task Force)**에 의해 개발되었다.
- TLS는 SSL의 후속 버전으로, **보안성**을 강화하고 **효율성**을 높인 프로토콜이다.
- TLS는 현재 널리 사용되는 보안 프로토콜로, 웹 브라우저, 이메일 클라이언트, VoIP 등 다양한 애플리케이션에서 사용된다.

## TLS의 발전 단계:

- **TLS 1.0**:
    - 1999년에 발표되었으며, SSL 3.0과 유사한 구조를 가지지만 보안 취약점이 보완되었다.
- **TLS 1.1**:
    - 2006년에 발표되었으며, 암호화 알고리즘이 개선되고 프로토콜의 여러 취약점이 보완되었다.
- **TLS 1.2**:
    - 2008년에 발표된 TLS의 주요 버전으로, 보다 강력한 **암호화 알고리즘**과 **해시 함수**가 도입되었으며, 현재까지도 널리 사용되고 있다.
- **TLS 1.3**:
    - 2018년에 발표된 최신 버전으로, 성능과 보안성이 크게 향상되었다.
    - 핸드셰이크 과정이 간소화되었으며, 불필요하거나 취약한 암호화 알고리즘을 제거하였다.

# 비교

| **특징** | **SSL** | **TLS** |
| --- | --- | --- |
| **개발 시기** | SSL 2.0 (1995년), SSL 3.0 (1996년) | TLS 1.0 (1999년), 최신 버전은 TLS 1.3 (2018년) |
| **암호화 강도** | 보안 취약점이 많음 | 보안성이 대폭 개선됨 |
| **보안 알고리즘** | MD5, SHA-1 등 취약한 해시 알고리즘 사용 | SHA-256 및 향상된 암호화 알고리즘 사용 |
| **핸드셰이크 과정** | 핸드셰이크가 상대적으로 복잡하고 느림 | 핸드셰이크가 간소화되고 효율성이 높아짐 |
| **세션 재개** | SSL은 세션 재개를 지원하지만 TLS보다 덜 효율적 | 세션 재개가 더 효율적 (TLS 1.2 이상에서 세션 재개 성능 향상) |
| **암호화 키 교환 방식** | RSA 주로 사용 | TLS는 Diffie-Hellman, ECDHE 등 다양한 키 교환 방식 지원 |
| **보안 취약점** | POODLE, DROWN 등 다양한 취약점 발견됨 | 보안성이 매우 강화되어 대부분의 취약점이 보완됨 |
| **프로토콜 지원 상태** | 대부분 시스템에서 사용 중단 | 현재 대부분의 보안 통신에서 사용됨 |
| **암호화 키 길이** | 40비트부터 시작해 현재는 128비트 이상 | 최소 128비트 이상을 사용하며, 256비트 암호화도 지원 |
| **핸드셰이크 메시지 인증** | 핸드셰이크 메시지 인증을 포함하지 않음 | 핸드셰이크 메시지에 인증을 추가하여 보안 강화 |

# **SSL과 TLS의 주요 기술적 차이**

### 1) **핸드셰이크 프로세스**:

- **SSL**:
    - 핸드셰이크 과정에서 데이터의 암호화 알고리즘을 협상하고, 인증서를 통해 클라이언트와 서버 간의 인증을 수행한다.
    - 이 과정이 복잡하고 시간이 걸릴 수 있다.
- **TLS**:
    - TLS에서는 핸드셰이크 과정이 더 간소화되었으며, 특히 **TLS 1.3**에서는 **RTT(Round Trip Time)**가 감소하여 연결 설정 속도가 빨라졌다.

### 2) **암호화 알고리즘**:

- **SSL**:
    - SSL에서는 주로 **MD5**, **SHA-1**과 같은 취약한 해시 함수가 사용되었으나, 이들은 이미 **보안 취약점**이 발견된 알고리즘이다.
- **TLS**:
    - TLS에서는 더 안전한 **SHA-256**과 같은 해시 함수와 **AES**와 같은 강력한 대칭 암호화 알고리즘을 사용한다.

### 3) **메시지 인증(MAC)**:

- **SSL**:
    - SSL에서는 메시지 인증을 수행할 때 메시지 다이제스트를 계산하기 위한 MAC(Message Authentication Code) 알고리즘이 존재하지만, 그 방식이 구식이다.
- **TLS**:
    - TLS에서는 **HMAC**(Hash-based Message Authentication Code)를 사용해 더 높은 보안성을 제공한다.

### 4) **키 교환 방식**:

- **SSL**:
    - SSL은 주로 **RSA** 기반의 키 교환을 사용하지만, 이 방식은 보안상 약점이 있다.
- **TLS**:
    - TLS는 **RSA**뿐만 아니라 **Diffie-Hellman**, **ECDHE**와 같은 **보다 안전한 키 교환 방식**을 지원하여 보안 수준을 높인다.

# **SSL과 TLS의 관계**

- **SSL**은 **TLS의 전신**으로, TLS는 SSL 3.0을 개선하여 보안성을 높인 프로토콜이다.
- **TLS 1.0**은 기본적으로 **SSL 3.0의 업그레이드 버전**으로 간주된다.
- **SSL**은 이제 더 이상 사용되지 않으며, **TLS**가 **사실상의 표준 보안 프로토콜**로 사용되고 있다.
- 현재 대부분의 웹 브라우저와 서버는 TLS를 지원하며, SSL은 호환성 유지를 위해 남아 있지만 **강력하게 사용이 권장되지 않는다**.

---

# **결론**

- **SSL**은 초기의 보안 프로토콜이었지만, 다양한 **보안 취약점**으로 인해 더 이상 사용되지 않고, SSL 3.0 이후 **TLS**가 이를 대체하게 되었다.
- **TLS**는 **더 강력한 암호화**, **간소화된 핸드셰이크 과정**, **다양한 키 교환 방식** 등을 통해 **보안성과 효율성**을 크게 개선한 프로토콜이다.
- 현재 보안 통신에서 **TLS**는 표준으로 사용되며, **SSL**은 더 이상 사용되지 않는 **구식 프로토콜**로 간주된다.
- TLS는 최신 버전인 **TLS 1.3**을 통해 더욱 **안전하고 효율적인 보안**을 제공하고 있다.
