## 개요

스타트업에서 인프라를 처음부터 구축할 때는 크게 "네트워크 기반 → 보안 → 컴퓨팅 → 데이터 → 배포 파이프라인 → 도메인/인증서" 순서로 진행하는 것이 일반적입니다. 아래는 AWS 기준으로 정리한 전체 구축 흐름입니다.

---

## 1단계: 네트워크 기반 (VPC)

### VPC 설계

가장 먼저 해야 할 일입니다. 나중에 바꾸기 가장 어려운 영역이기 때문에 처음에 신중하게 설계해야 합니다.

- CIDR 블록 결정 (ex: `10.0.0.0/16`)
- AZ(가용영역) 몇 개를 쓸지 결정 (최소 2개 권장)
- 환경 분리 전략: dev/staging/prod를 VPC로 분리할지, namespace로만 분리할지

### Subnet 구성

| 종류 | 용도 | 예시 CIDR |
| --- | --- | --- |
| Public Subnet | Bastion, NAT GW, ALB | `10.0.1.0/24` |
| Private Subnet | EKS 노드, 앱 서버 | `10.0.10.0/24` |
| DB Subnet | RDS, ElastiCache | `10.0.20.0/24` |

### Internet Gateway / NAT Gateway

- "IGW": Public Subnet에서 인터넷 직접 연결
- "NAT GW": Private Subnet에서 아웃바운드 인터넷 연결 (인바운드는 차단)
- NAT GW는 AZ마다 하나씩 두는 게 HA 구성 (비용과 트레이드오프)

---

## 2단계: 보안 (ACG / Security Group / ACL)

### Security Group (ACG)

리소스 단위 방화벽입니다. 상태 저장(Stateful) 방식으로, 인바운드 허용 시 아웃바운드 응답은 자동 허용됩니다.

- Bastion SG: `22/tcp` 인바운드, 출처를 내 IP로만 제한
- EKS Node SG: Bastion에서만 SSH 허용, ALB에서 NodePort 허용
- RDS SG: EKS Node SG에서만 DB 포트 허용
- ALB SG: `80`, `443` 전체 인바운드

### Network ACL

Subnet 단위 방화벽입니다. 상태 비저장(Stateless) 방식으로, 인바운드/아웃바운드 규칙을 모두 명시해야 합니다. SG만으로도 충분히 운영 가능하며, NACL은 서브넷 레벨 추가 방어선으로 활용합니다.

---

## 3단계: 컴퓨팅

### Bastion Host

- Public Subnet에 배치하는 EC2
- 외부에서 Private 리소스로 접근하는 유일한 진입점
- 최소한의 사이즈로 운영 (t3.micro 수준)
- Session Manager(SSM)으로 대체하면 Bastion 자체를 없앨 수도 있음

### Kubernetes (EKS vs Self-managed)

| 항목 | EKS | Self-managed (Bastion 내) |
| --- | --- | --- |
| 관리 부담 | 낮음 | 높음 |
| 비용 | 클러스터 비용 추가 | EC2 비용만 |
| HA | 기본 제공 | 직접 구성 |
| 권장 시점 | 대부분 | PoC/초소형 |

스타트업 초기에도 EKS를 권장합니다. 운영 부담 차이가 크기 때문입니다.

### EKS 구성 요소

- Node Group: EC2 타입/사이즈, Auto Scaling 설정
- IRSA (IAM Roles for Service Accounts): Pod에 IAM 권한 부여
- EKS Add-on: CoreDNS, kube-proxy, VPC CNI, EBS CSI Driver

---

## 4단계: 데이터베이스

### RDS vs EC2 내 DB

| 항목 | RDS | EC2 직접 설치 |
| --- | --- | --- |
| 자동 백업 | ✅ | ❌ |
| Multi-AZ | ✅ (옵션) | 직접 구성 |
| 운영 부담 | 낮음 | 높음 |
| 비용 | 높음 | 낮음 |
| 권장 | 대부분 | 비용 극한 절감 시 |

스타트업 초기에는 RDS 단일 인스턴스로 시작하고, 서비스 안정화 후 Multi-AZ 전환하는 패턴이 일반적입니다.

### DB Subnet Group

RDS는 반드시 "DB Subnet Group"을 별도로 생성해서 Private Subnet에 배치해야 합니다.

## 5단계: Container Registry

### 옵션 비교

| 서비스 | 특징 |
| --- | --- |
| AWS ECR | IAM 연동 쉬움, EKS와 궁합 좋음 |
| Docker Hub | 무료 플랜 제한 있음 |
| GitHub Container Registry | GitHub Actions와 연동 편리 |
| Self-hosted Harbor | 완전한 제어, 운영 부담 있음 |

AWS 환경이면 ECR이 가장 자연스럽습니다. Private ECR + IRSA 조합으로 인증 없이 Pull 가능하게 구성합니다.

---

## 6단계: CI/CD 파이프라인

### Jenkins

- EC2에 직접 설치하거나 EKS 내 Pod으로 운영
- EKS 내 운영 시 동적 Agent (Pod Template) 사용 가능
- ECR Push 권한: IRSA 또는 IAM User Access Key

### ArgoCD

- EKS 클러스터 내 설치
- Git 저장소의 변경을 감지해 클러스터에 자동 반영
- App of Apps 패턴으로 여러 서비스 한꺼번에 관리
- RBAC 설정으로 팀원별 접근 제어

### 전체 파이프라인 흐름

```
코드 Push → Jenkins (빌드/테스트/ECR Push) → manifest repo 업데이트 → ArgoCD Sync → EKS 배포
```

## 7단계: 도메인 / 인증서 / 인그레스

### 도메인

- Route 53에서 도메인 구매하거나 외부 도메인을 Route 53으로 NS 위임
- 서비스별 서브도메인 전략 사전 정의 (ex: `api.example.com`, `admin.example.com`)

### 인증서 (TLS/SSL)

- AWS ACM (Certificate Manager): 무료 발급, ALB에 붙이는 방식으로 사용
- cert-manager: EKS 내에서 Let's Encrypt 인증서 자동 발급/갱신

### ALB Ingress Controller

- EKS에서 ALB를 Ingress로 사용하려면 "AWS Load Balancer Controller" 설치 필요
- Ingress 오브젝트 → ALB 자동 생성

---

## 8단계: 관측성 (Observability)

인프라 구축 후 바로 이어서 해야 할 항목입니다.

### 최소 구성

- "로깅": CloudWatch Logs 또는 Loki + Grafana
- "메트릭": Prometheus + Grafana
- "트레이싱": 초기엔 생략 가능, 이후 Jaeger/Tempo 도입

---

## 전체 순서 요약

```
1. VPC / Subnet 설계
2. IGW / NAT GW
3. Security Group / NACL
4. Bastion EC2
5. EKS 클러스터 + Node Group
6. RDS (DB Subnet Group)
7. ECR (Container Registry)
8. Jenkins 구축
9. ArgoCD 설치
10. ALB Ingress Controller
11. 도메인 (Route 53)
12. 인증서 (ACM / cert-manager)
13. Observability (Prometheus, Grafana, Loki)
```

네트워크가 가장 먼저이고 관측성이 가장 나중이지만, 실제로는 병렬로 진행되는 항목도 많습니다. 특히 Jenkins와 ArgoCD는 EKS 구성 직후 바로 이어가는 것이 효율적입니다.
