## 개요

git flow의 기본 구조와 `main > stg > dev` 3단계 브랜치 전략을 결합한 운영 방식입니다.

git flow는 메인 브랜치를 2개만 지원하므로, 3단계 구조를 완벽하게 수용할 수 없습니다. 이를 해결하기 위해 아래 설정과 운영 방식을 사용합니다.

---

## git flow init 설정

```
Branch name for production releases: main
Branch name for "next release" development: dev
Feature branches: feature/
Release branches: release/
Hotfix branches: hotfix/
Version tag prefix: v
```

핵심은 `next release`를 `dev`로 설정하는 것입니다.

`stg`는 git flow가 인식하지 못하는 브랜치이므로 수동으로 생성하고 관리합니다.

```bash
git checkout -b stg
git push origin stg
```

---

## 브랜치 역할

| 브랜치 | 환경 | 목적 |
| --- | --- | --- |
| main | Production | 실제 서비스 운영 |
| stg | Staging | QA / 운영 전 최종 검증 |
| dev | Development | 개발자 통합 테스트 |

---

## 전체 흐름

```
feature/* → dev → stg → main
hotfix/*  → main → stg → dev (역병합)
```

---

## 기능 개발 사이클

`git flow feature start/finish` 사용

```bash
# dev 기준으로 자동 분기
git flow feature start my-feature

# 작업 완료 후 dev로 자동 병합
git flow feature finish my-feature
```

`next release = dev` 설정 덕분에 feature 브랜치가 자동으로 `dev` 기준으로 분기/병합됩니다.

---

## 배포 사이클

`git flow release start/finish` 사용

```bash
# 1. dev 내용을 stg에 반영
git checkout stg
git merge dev
# QA 진행

# 2. QA 완료 후 release 시작 (dev 기준으로 분기됨)
git flow release start 1.0.0

# 3. release 완료 → main 병합 + v1.0.0 태그 자동 생성
git flow release finish 1.0.0
```

`release start`는 `dev` 기준으로 분기되므로, stg에서 QA한 내용이 dev에 반영되어 있어야 release 내용이 일치합니다.

따라서 순서는 `dev → stg merge`이고, 반대 방향이 아닙니다.

---

## git flow 명령어 사용 여부

| 명령어 | 사용 여부 | 이유 |
| --- | --- | --- |
| feature start/finish | ✅ | dev 기준 자동 분기/병합 |
| release start/finish | ✅ | main 병합 + 태그 자동 생성 |
| hotfix start/finish | ✅ | main 기준 긴급 수정, dev 수동 동기화 필요 |

---

## hotfix 후 dev 동기화

`hotfix finish`는 `main`과 `dev`에만 병합하고, `stg`는 자동 동기화되지 않습니다.

```bash
git flow hotfix finish fix-issue

# stg 수동 동기화
git checkout stg
git merge main
git push origin stg
```

---

## 한 줄 요약

- 개발 사이클 → dev에서 `git flow feature start/finish`
- 배포 사이클 → stg에서 `merge dev` → `git flow release start/finish`
