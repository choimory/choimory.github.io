# 웹소켓과 HTTP의 차이

- 웹소켓(WebSocket)과 HTTP는 네트워크 통신을 위한 프로토콜이지만, 둘 사이에는 작동 방식과 사용 목적에서 중요한 차이가 있다.
- 각 프로토콜은 특정 요구 사항에 맞게 설계되어 있으며, 사용되는 방식과 기능에 차이가 있다.

---

# **HTTP**

- **(통신 방식)** 단방향 통신(요청/응답 기반). 클라이언트가 요청을 보내면 서버가 응답.
- **(연결 유지)** 요청/응답이 완료되면 연결 종료. `Keep-Alive` 옵션으로 연결 유지 가능.
- **(프로토콜 특성)** 상태 비저장(Stateless). 각 요청은 독립적이며, 상태를 저장하지 않음.
- **(오버헤드)** 요청마다 헤더가 포함되어 있어 오버헤드가 큼.
- **(데이터 전송)** 주로 텍스트 기반의 데이터 전송.
- **(사용 사례)** 웹 페이지 로딩, 파일 다운로드, 데이터베이스 조회 등 주로 정적인 서비스에 사용.
- **(프로토콜 업그레이드)** 기본적으로 고정된 요청/응답 패턴.

---

# **웹소켓(WebSocket)**

- **(통신 방식)** 양방향 통신. 클라이언트와 서버가 실시간으로 데이터를 주고받을 수 있음.
- **(연결 유지)** 연결이 설정되면 클라이언트와 서버 간의 지속적인 연결을 유지.
- **(프로토콜 특성)** 상태 저장(Stateful). 연결 동안 클라이언트와 서버의 상태가 유지됨.
- **(오버헤드)** 초기 핸드셰이크 이후 오버헤드가 거의 없음.
- **(데이터 전송)** 텍스트와 바이너리 데이터 모두 전송 가능.
- **(사용 사례)** 실시간 채팅, 온라인 게임, 실시간 알림, 주식 거래 시스템 등 상호작용이 필요한 서비스에 사용.
- **(프로토콜 업그레이드)** HTTP 요청에서 시작하여 `Upgrade` 헤더를 통해 웹소켓으로 전환.

---

# 비교

| **특징** | **HTTP** | **웹소켓(WebSocket)** |
| --- | --- | --- |
| **통신 방식** | 단방향 (요청/응답 기반) | 양방향 (실시간 상시 연결) |
| **연결 유지** | 요청/응답 후 연결 종료 | 연결을 유지하고 지속적인 데이터 송수신 가능 |
| **프로토콜 특성** | 상태 비저장 (Stateless) | 상태 저장 (Stateful) |
| **오버헤드** | 매 요청마다 헤더 포함, 오버헤드 큼 | 초기에만 HTTP 오버헤드, 이후 헤더 없음 |
| **사용 사례** | 웹 페이지 로딩, 파일 다운로드 등 | 실시간 채팅, 게임, 알림, 주식 거래 등 |
| **데이터 전송** | 주로 텍스트 기반 | 텍스트 및 바이너리 데이터 전송 가능 |

---

# 결론

- **HTTP**는 **요청/응답 기반의 단방향 통신**으로, 주로 정적인 콘텐츠 전송에 적합하다.
- **웹소켓**은 **양방향 실시간 통신**을 위한 프로토콜로, 상시 연결을 유지하며 서버와 클라이언트 간 **실시간 상호작용**이 필요한 경우에 사용된다.
