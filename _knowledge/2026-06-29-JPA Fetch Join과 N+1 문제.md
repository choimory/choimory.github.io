# JPA Fetch Join

## 개요

JPA에서 연관 엔티티를 조회할 때 N+1 문제는 가장 흔하게 마주치는 성능 이슈입니다. Fetch Join은 이 문제를 해결하기 위해 JPQL에서 제공하는 기능으로, 연관된 엔티티를 한 번의 쿼리로 함께 조회할 수 있게 해줍니다. 단순히 "조인을 해서 가져온다"는 의미를 넘어서, 영속성 컨텍스트와 SQL 쿼리 생성 방식에 미치는 영향을 정확히 이해하는 것이 중요합니다.

---

## N+1 문제와 Fetch Join의 등장 배경

### N+1 문제란

연관관계가 설정된 엔티티를 조회할 때, 처음 조회한 엔티티 개수(N)만큼 연관 엔티티를 조회하는 추가 쿼리가 발생하는 현상입니다.

#### 발생 시나리오

"Team"을 조회한 후 각 "Team"에 속한 "Member" 리스트를 LAZY 로딩으로 접근하면, "Team"을 조회하는 쿼리 1번에 더해 각 "Team"마다 "Member"를 조회하는 쿼리가 추가로 발생합니다. "Team"이 10개라면 총 11번의 쿼리(1 + 10)가 실행되는 것입니다.

#### EAGER로 해결되지 않는 이유

연관관계를 EAGER로 설정해도 문제는 해결되지 않습니다. JPQL로 조회하는 경우 JPA는 JPQL을 먼저 SQL로 그대로 번역한 뒤, 연관 엔티티가 EAGER 설정이라는 것을 보고 각각을 추가로 조회하는 쿼리를 별도로 실행하기 때문입니다. 즉 N+1 문제는 FetchType 설정과 무관하게, "연관 엔티티를 별도 쿼리로 가져오는 구조" 자체에서 발생합니다.

### Fetch Join의 필요성

Fetch Join은 JPQL 단계에서 명시적으로 "이 연관 엔티티는 조인해서 함께 가져와라"라고 지정하는 방법입니다. 이렇게 하면 SQL 자체가 JOIN으로 생성되어 한 번의 쿼리로 모든 데이터를 가져올 수 있습니다.

---

## Fetch Join의 동작 방식

### 기본 문법

```java
@Query("SELECT t FROM Team t JOIN FETCH t.members WHERE t.id = :teamId")
Team findTeamWithMembers(@Param("teamId") Long teamId);
```

일반 JOIN과 다르게 "FETCH" 키워드가 추가되는 것이 핵심입니다.

#### 일반 JOIN과의 차이

일반 JPQL JOIN은 조인 조건으로 필터링만 할 뿐, SELECT 절에 명시되지 않은 연관 엔티티는 함께 조회하지 않습니다. 즉 "SELECT t FROM Team t JOIN t.members m"처럼 작성하면 SQL은 JOIN으로 나가지만, "Team"의 "members" 컬렉션은 여전히 프록시 상태로 남아 실제 접근 시 추가 쿼리가 발생합니다. 반면 Fetch Join은 연관 엔티티의 실제 데이터까지 SELECT 절에 포함시켜 함께 영속성 컨텍스트에 적재합니다.

### 영속성 컨텍스트에 미치는 영향

Fetch Join으로 조회된 연관 엔티티는 프록시가 아닌 실제 객체로 영속성 컨텍스트에 저장됩니다. 따라서 이후 해당 연관 엔티티에 접근할 때 추가 쿼리 없이 즉시 사용할 수 있고, 지연 로딩(LAZY) 설정 여부와 무관하게 즉시 로딩된 것처럼 동작합니다.

#### 글로벌 fetch 전략과의 관계

Fetch Join은 엔티티에 설정된 "fetch = FetchType.LAZY" 같은 글로벌 설정을 무시하고, 해당 JPQL 쿼리 한 번에 한해서만 즉시 로딩으로 동작하게 만듭니다. 엔티티 매핑 자체를 변경하는 것이 아니라 특정 쿼리에서만 일시적으로 즉시 로딩 전략을 덮어쓰는 것입니다.

---

## Fetch Join 사용 시 주의사항

### 컬렉션 Fetch Join과 페이징

컬렉션("@OneToMany")을 Fetch Join하면서 페이징("Pageable")을 함께 사용하면 문제가 발생합니다. JPA는 컬렉션 Fetch Join 시 데이터베이스 레벨의 LIMIT/OFFSET을 적용하지 못하고, 전체 데이터를 메모리에 로드한 후 메모리에서 페이징을 처리합니다. 데이터가 많을 경우 메모리 부족이나 심각한 성능 저하로 이어질 수 있습니다.

#### 해결 방법

이 문제를 회피하는 대표적인 방법은 다음과 같습니다.

"@BatchSize"를 사용하는 방법은 컬렉션을 Fetch Join하지 않고 일반 LAZY 로딩으로 두되, "hibernate.default_batch_fetch_size" 설정이나 "@BatchSize(size = N)" 어노테이션을 통해 N개씩 IN 쿼리로 묶어서 가져오는 방식입니다. 이렇게 하면 N+1 문제를 완화하면서도 페이징은 정상적으로 데이터베이스에서 처리됩니다.

"ToOne 관계만 Fetch Join"하는 방법은 페이징이 필요한 쿼리에서는 "@ManyToOne"이나 "@OneToOne" 관계만 Fetch Join하고, "@OneToMany" 컬렉션은 별도로 조회하거나 지연 로딩에 맡기는 방식입니다. ToOne 관계는 컬렉션이 아니므로 페이징에 영향을 주지 않습니다.

### 둘 이상의 컬렉션 Fetch Join 제한

하나의 JPQL에서 둘 이상의 컬렉션을 동시에 Fetch Join하면 "MultipleBagFetchException"이 발생할 수 있습니다. 이는 두 컬렉션을 카티전 곱(Cartesian Product) 형태로 조인하면서 데이터 정합성을 보장할 수 없기 때문입니다.

#### 해결 방법

컬렉션 타입을 "List" 대신 "Set"으로 변경하면 예외는 피할 수 있지만, 카티전 곱으로 인한 중복 데이터와 데이터 폭증 문제는 여전히 남습니다. 일반적으로는 컬렉션 중 하나만 Fetch Join하고 나머지는 별도 쿼리나 "@BatchSize"로 처리하는 것이 더 안전합니다.

### DISTINCT 키워드의 필요성

"@OneToMany" Fetch Join 시 데이터베이스 조인 특성상 부모 엔티티가 자식 엔티티 수만큼 중복되어 조회됩니다. JPQL에 "DISTINCT"를 추가하면 애플리케이션 레벨에서 중복된 부모 엔티티를 제거해줍니다. 다만 SQL 자체의 DISTINCT와는 다르게, JPA는 SQL 결과를 애플리케이션에서 한 번 더 중복 제거하는 방식으로 동작합니다.

---

## Spring Data JPA에서의 활용

### @EntityGraph

"@EntityGraph"는 Fetch Join을 어노테이션 기반으로 선언할 수 있게 해주는 기능입니다.

```kotlin
@EntityGraph(attributePaths = ["members"])
@Query("SELECT t FROM Team t WHERE t.id = :teamId")
fun findTeamWithMembers(@Param("teamId") teamId: Long): Team?
```

JPQL을 직접 작성하지 않고도 "attributePaths"에 명시한 연관관계를 Fetch Join 형태로 동작시킬 수 있어, 메서드 이름 기반 쿼리("findBy...")에도 Fetch Join을 적용할 수 있다는 장점이 있습니다.

### Kotlin에서의 사용

Kotlin에서는 JPQL 문자열 자체는 Java와 동일하지만, "@Param"과 nullable 반환 타입을 명시적으로 다루는 것이 일반적입니다. 엔티티를 찾지 못할 가능성이 있는 단건 조회는 "Team?"처럼 nullable로 선언하고, 호출부에서 "?:" 엘비스 연산자나 커스텀 예외로 null을 처리하는 패턴이 자주 쓰입니다.

---

## 전체 요약

Fetch Join은 JPQL에서 "JOIN FETCH" 키워드를 사용해 연관 엔티티를 한 번의 SQL 쿼리로 함께 조회함으로써 N+1 문제를 해결하는 기능입니다. 일반 JOIN과 달리 연관 엔티티를 실제 객체로 영속성 컨텍스트에 적재하며, 엔티티에 설정된 글로벌 FetchType과 무관하게 해당 쿼리에서만 즉시 로딩처럼 동작합니다. 다만 컬렉션을 Fetch Join하면서 페이징을 함께 쓰면 메모리에서 페이징이 처리되는 문제가 있어 "@BatchSize"나 ToOne 관계만 Fetch Join하는 방식으로 회피해야 하고, 둘 이상의 컬렉션을 동시에 Fetch Join하면 "MultipleBagFetchException"이 발생할 수 있다는 제약도 있습니다. Spring Data JPA에서는 "@EntityGraph"를 통해 어노테이션만으로 Fetch Join을 선언할 수도 있습니다.

# Fetch Join의 SQL 동작 방식

## 개요

"Bag" 1개에 "Item" 10개가 연관되어 있을 때 Fetch Join을 사용하면, JPA는 실제로 "Bag"과 "Item"을 INNER JOIN으로 묶어서 한 번에 조회합니다. 다만 이때 SQL 결과 row 수와 JPA가 최종적으로 영속성 컨텍스트에 적재하는 객체 수 사이에 차이가 발생하는데, 이 부분을 정확히 이해하는 것이 중요합니다.

---

## SQL 레벨에서의 동작

### 실제 생성되는 쿼리

```java
@Query("SELECT b FROM Bag b JOIN FETCH b.items WHERE b.id = :bagId")
Bag findBagWithItems(@Param("bagId") Long bagId);
```

이 JPQL은 다음과 비슷한 SQL로 번역됩니다.

```sql
SELECT b.*, i.*
FROM bag b
INNER JOIN item i ON b.id = i.bag_id
WHERE b.id = ?
```

"Bag" 1개와 "Item" 10개가 INNER JOIN으로 묶이면, SQL 결과는 10개의 row가 반환됩니다. 각 row에는 동일한 "Bag" 컬럼 값이 10번 반복되고, 그 옆에 서로 다른 "Item" 컬럼 값이 붙어 있는 형태입니다.

### Row 중복이 발생하는 이유

이는 일반적인 RDBMS의 JOIN 특성 때문입니다. "1:N" 관계를 JOIN하면 "1" 쪽 데이터가 "N" 쪽 개수만큼 반복되어 나오는 것이 당연한 동작이며, JPA가 별도로 처리하지 않는 한 이는 순수 SQL 차원의 결과입니다.

---

## JPA가 row를 객체로 변환하는 과정

### 영속성 컨텍스트에서의 중복 제거

JPA는 SQL 결과 10개 row를 그대로 10개의 "Bag" 객체로 만들지 않습니다. 대신 각 row에서 "Bag"의 식별자(PK)를 확인하고, 영속성 컨텍스트에 이미 동일한 식별자의 "Bag"이 있는지 먼저 검사합니다.

#### 동일성 보장 메커니즘

첫 번째 row를 처리할 때 "Bag" 객체를 생성해서 영속성 컨텍스트에 등록하고, 두 번째 row부터는 동일한 PK를 가진 "Bag"이 이미 있으므로 새로 생성하지 않고 기존 "Bag" 객체를 재사용합니다. 대신 그 "Bag" 객체의 "items" 컬렉션에 각 row의 "Item"을 하나씩 추가해 나갑니다. 결과적으로 최종 반환값은 "Item" 10개를 담은 "Bag" 객체 1개가 됩니다.

이것이 가능한 이유는 영속성 컨텍스트가 "같은 트랜잭션 내에서 동일한 PK를 가진 엔티티는 항상 동일한 객체 인스턴스"라는 동일성(Identity) 원칙을 보장하기 때문입니다.

### List가 아닌 Set을 쓸 때의 차이

만약 "Bag.items"가 "List"라면 JPA는 별다른 중복 제거 없이 row 순서대로 10개를 그대로 추가합니다. 반면 "Set"이라면 컬렉션 자체의 특성상 중복 원소가 들어갈 수 없지만, 이 경우 "Item"들은 서로 다른 PK를 가지므로 애초에 중복될 일이 없어 결과는 동일합니다. "DISTINCT"가 의미를 가지는 경우는 컬렉션이 둘 이상 얽혀 카티전 곱이 발생할 때나, "Bag" 자체가 중복으로 잡힐 때입니다.

---

## 단일 컬렉션 Fetch Join에서 DISTINCT가 불필요한 이유

"Bag" 1개에 "Item" 컬렉션 하나만 Fetch Join하는 경우에는 "Bag" 객체가 중복 생성되는 문제가 없습니다. 영속성 컨텍스트의 동일성 보장 메커니즘이 이미 중복을 걸러주기 때문입니다. 따라서 "DISTINCT"는 보통 컬렉션이 2개 이상 동시에 Fetch Join 되어 카티전 곱이 발생하는 상황에서 필요해집니다.

---

## 전체 요약

"Bag" 1개와 "Item" 10개를 Fetch Join하면 SQL 레벨에서는 정확히 INNER JOIN으로 실행되며, 그 결과 10개의 row가 반환됩니다. 각 row에는 동일한 "Bag" 데이터가 반복되지만, JPA는 영속성 컨텍스트의 동일성 보장 메커니즘을 통해 PK가 같은 "Bag"을 재사용하면서 각 row의 "Item"만 컬렉션에 누적시켜, 최종적으로는 "Item" 10개를 가진 "Bag" 객체 1개로 변환합니다. 이 과정에서 row 중복 제거는 애플리케이션(JPA) 레벨에서 일어나는 것이며, SQL 자체가 중복을 줄여주는 것은 아닙니다.

---

# Fetch Join INNER JOIN의 성능 부담

## 개요

"INNER JOIN이 무겁지 않냐"는 질문은 두 가지 다른 층위로 나눠서 봐야 합니다. 하나는 "JOIN 연산 자체가 데이터베이스 입장에서 비용이 드는가"이고, 다른 하나는 "Fetch Join으로 인한 row 중복이 전송량과 메모리를 늘려서 부담이 되는가"입니다. 둘을 구분하지 않으면 막연히 "JOIN은 무겁다"는 인상만 남게 되는데, 실제로는 N+1과 비교했을 때 대부분의 경우 Fetch Join이 더 가볍습니다.

---

## JOIN 연산 자체의 비용

### 인덱스가 있는 경우

"Item" 테이블의 "bag_id"처럼 외래키 컬럼에 인덱스가 걸려 있다면, INNER JOIN은 일반적으로 인덱스를 통한 효율적인 탐색(Index Seek)으로 처리됩니다. "@ManyToOne"/"@OneToMany" 관계의 외래키는 보통 인덱스가 자동으로 생성되거나 직접 추가하는 것이 일반적이므로, 단순 INNER JOIN 자체는 데이터베이스 옵티마이저가 잘 최적화하는 영역입니다.

### 인덱스가 없거나 대용량일 때

인덱스가 없는 컬럼으로 조인하거나 양쪽 테이블이 매우 크다면 풀 스캔이나 해시 조인 등 비용이 큰 실행 계획이 선택될 수 있습니다. 하지만 이는 Fetch Join에 한정된 문제가 아니라 일반적인 SQL JOIN 전반에 해당하는 데이터베이스 튜닝의 영역입니다.

---

## Fetch Join 특유의 부담: row 중복

### 전송량 증가

"Bag" 1개와 "Item" 10개를 Fetch Join하면 "Bag"의 모든 컬럼 값이 10번 반복되어 네트워크로 전송됩니다. "Bag"에 큰 텍스트 컬럼이나 많은 필드가 있다면 이 중복 전송량이 누적되어 실제 부담으로 작용할 수 있습니다. 컬렉션 크기가 매우 크거나(수백~수천 건) 부모 엔티티의 컬럼이 무거울 때 이 효과가 두드러집니다.

### N+1과의 비교

N+1로 처리하면 "Bag" 조회 1번 + "Item" 조회 1번(또는 N번), Fetch Join은 조회 1번이지만 row가 10개로 부풀려집니다. 일반적으로 쿼리 횟수를 줄이는 것이 네트워크 라운드트립(round trip) 비용을 줄이는 효과가 더 크기 때문에, row 중복으로 인한 데이터량 증가보다 쿼리 횟수 감소의 이득이 더 큰 경우가 많습니다. 다만 이는 "보통 그렇다"는 일반론이고, 실제로는 컬렉션 크기와 부모 엔티티 크기에 따라 달라지는 트레이드오프입니다.

---

## 부담이 실제로 커지는 상황

### 컬렉션이 매우 크거나 여러 단계로 중첩될 때

"Bag" → "Item" → "ItemDetail"처럼 여러 단계를 한꺼번에 Fetch Join하면 각 단계마다 row가 곱연산으로 불어납니다. "Item"이 10개, "ItemDetail"이 각각 5개라면 최종 row 수는 50개가 됩니다. 이런 깊은 체이닝은 데이터 폭증으로 직결되므로 주의가 필요합니다.

### 페이징과 함께 쓸 때

컬렉션 Fetch Join과 페이징을 함께 쓰면 데이터베이스가 아닌 애플리케이션 메모리에서 페이징을 처리하게 되어, 이 경우는 단순 row 중복보다 훨씬 심각한 부담(전체 데이터 메모리 로딩)으로 이어집니다.

---

## 결론: 무거운가 가벼운가

단순 INNER JOIN 연산 자체는 인덱스가 적절히 걸려 있다면 무거운 작업이 아닙니다. 진짜 부담은 "row 중복으로 인한 데이터 전송량 증가"이고, 이는 컬렉션 크기와 부모 엔티티의 무게에 비례해서 커집니다. 대부분의 경우 이 부담은 N+1로 여러 번 쿼리를 날리는 것보다 작지만, 컬렉션이 크거나 여러 단계로 중첩되면 오히려 Fetch Join이 더 무거워질 수 있습니다.

---

## 전체 요약

Fetch Join의 INNER JOIN 자체는 인덱스가 잘 잡혀 있다면 데이터베이스 입장에서 비용이 크지 않은 연산입니다. 실제 부담은 1:N 관계 특성상 발생하는 row 중복에서 오며, 이는 부모 엔티티의 컬럼 데이터가 자식 개수만큼 반복 전송되는 형태로 나타납니다. 일반적으로는 쿼리 횟수를 줄이는 이득이 row 중복으로 인한 데이터량 증가보다 크지만, 컬렉션이 매우 크거나 여러 단계로 중첩되거나 페이징과 함께 사용될 때는 오히려 Fetch Join이 더 무거운 선택이 될 수 있어 상황에 맞게 판단해야 합니다.

---

# @BatchSize를 활용한 컬렉션 배치 조회

## 개요

"@BatchSize"는 Fetch Join처럼 한 번의 JOIN으로 모든 데이터를 가져오는 대신, 지연 로딩(LAZY)을 유지하면서도 N+1 문제를 완화하는 또 다른 접근 방식입니다. 핵심 아이디어는 "각 부모마다 개별 쿼리를 날리지 말고, 여러 부모의 자식들을 IN 절로 한꺼번에 모아서 가져온다"는 것입니다. Fetch Join과 비교했을 때 쿼리 횟수와 데이터 전송량 측면에서 뚜렷한 차이를 보입니다.

---

## @BatchSize의 동작 방식

### 기본 설정

```kotlin
@Entity
class Bag(
    @Id @GeneratedValue
    val id: Long? = null,

    @OneToMany(mappedBy = "bag", fetch = FetchType.LAZY)
    @BatchSize(size = 100)
    val items: MutableList<Item> = mutableListOf()
)
```

또는 전역 설정으로 모든 컬렉션에 일괄 적용할 수도 있습니다.

```yaml
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 100
```

### 실제 쿼리 동작

"Bag"을 10개 조회한 뒤 각 "Bag"의 "items"에 접근하면, "@BatchSize"가 없을 때는 "Bag"마다 개별적으로 "SELECT * FROM item WHERE bag_id = ?"가 10번 실행됩니다. "@BatchSize(size = 100)"이 설정되어 있으면, 첫 번째 "Bag"의 "items"에 접근하는 순간 Hibernate가 나머지 9개 "Bag"의 ID까지 한꺼번에 모아서 다음과 같은 쿼리를 한 번만 실행합니다.

```sql
SELECT * FROM item WHERE bag_id IN (?, ?, ?, ?, ?, ?, ?, ?, ?, ?)
```

"size"보다 부모 엔티티 수가 많다면 그 수만큼 나눠서 여러 번의 IN 쿼리가 발생합니다. 예를 들어 "Bag"이 250개이고 "size = 100"이라면 IN 쿼리가 3번(100+100+50) 발생합니다.

---

## 쿼리 횟수 비교

### Fetch Join

```sql
SELECT b.*, i.* FROM bag b INNER JOIN item i ON b.id = i.bag_id
```

쿼리 1번으로 모든 데이터를 가져오지만, "Bag"이 10개이고 각 "Bag"마다 "Item"이 10개라면 결과 row는 100개가 되고, 각 row마다 "Bag"의 모든 컬럼이 반복 전송됩니다.

### @BatchSize

```sql
-- 1번째 쿼리
SELECT * FROM bag;
-- 2번째 쿼리 (items 접근 시 한 번에 묶어서)
SELECT * FROM item WHERE bag_id IN (...);
```

쿼리는 총 2번(1 + 1)이 발생하지만, 각 쿼리의 결과 row는 중복 없이 깔끔합니다. "Bag" 10개 row, "Item" 100개 row로, 전송되는 "Bag" 컬럼 데이터양이 Fetch Join보다 훨씬 적습니다.

---

## 데이터 전송량 비교

### Fetch Join의 중복 전송 문제

"Bag"에 컬럼이 많거나 큰 텍스트 필드("description" 같은)가 있다면, Fetch Join은 그 컬럼 값을 자식 개수만큼 반복해서 네트워크로 전송합니다. "Bag" 1개에 "Item"이 100개라면 "Bag"의 무거운 컬럼이 100번 중복 전송되는 것입니다.

### @BatchSize는 중복이 없음

"@BatchSize"는 "Bag"과 "Item"을 별도 쿼리로 분리하기 때문에 "Bag" 데이터는 정확히 "Bag" 개수만큼만, "Item" 데이터는 정확히 "Item" 개수만큼만 전송됩니다. 구조적으로 중복 전송이 발생하지 않습니다.

---

## 페이징과의 호환성

### Fetch Join의 한계

컬렉션을 Fetch Join하면서 "Pageable"을 함께 사용하면 데이터베이스 LIMIT/OFFSET이 적용되지 않고, 전체 데이터를 메모리에 올린 뒤 애플리케이션에서 페이징을 처리하게 됩니다.

### @BatchSize는 페이징과 자연스럽게 호환

"@BatchSize"는 부모 엔티티 조회 자체가 일반 LAZY 로딩이므로, "Pageable"을 그대로 사용해도 데이터베이스 레벨에서 정상적으로 LIMIT/OFFSET이 적용됩니다. 컬렉션은 부모를 다 가져온 뒤 별도로 배치 조회되므로 페이징과 충돌이 없습니다.

```kotlin
fun findAll(pageable: Pageable): Page<Bag>
```

이 메서드를 그대로 써도 "Bag"은 정상적으로 페이징되고, 이후 "items"에 접근하는 시점에 "@BatchSize"가 적용됩니다.

---

## 어떤 상황에서 무엇을 선택할지

### Fetch Join이 유리한 경우

연관 엔티티가 ToOne 관계이거나, 컬렉션이라도 크기가 작고 페이징이 필요 없는 단건/소량 조회일 때는 쿼리 1번으로 끝내는 Fetch Join이 더 간결하고 유리합니다.

### @BatchSize가 유리한 경우

컬렉션 크기가 크거나, 페이징이 필요하거나, 부모 엔티티의 컬럼이 무거워서 중복 전송 부담이 큰 경우에는 "@BatchSize"가 더 적합합니다. 또한 둘 이상의 컬렉션을 동시에 다뤄야 해서 "MultipleBagFetchException"이 우려되는 상황에서도, 각 컬렉션마다 "@BatchSize"를 적용하면 카티전 곱 문제 자체가 발생하지 않습니다.

---

## 전체 요약

"@BatchSize"는 Fetch Join과 달리 부모와 자식을 별도의 쿼리로 분리하되, 자식 조회 시 여러 부모의 ID를 IN 절로 묶어서 한 번에 가져오는 방식으로 N+1을 완화합니다. 쿼리 횟수는 Fetch Join(1번)보다 많지만(부모 1번 + 자식 N/size번), row 중복이 없어 데이터 전송량이 적고 페이징과도 자연스럽게 호환됩니다. 반면 Fetch Join은 쿼리 횟수가 가장 적지만 1:N 관계의 row 중복으로 데이터 전송량이 늘어나고 페이징과 함께 쓰기 어렵습니다. 결국 컬렉션 크기, 페이징 필요 여부, 부모 엔티티의 무게를 기준으로 두 전략 중 하나를 선택해야 합니다.

---

# 다단계 연관관계에서 @BatchSize 적용 전략

## 개요

"Bag" → "Item" → "ItemDetail"처럼 컬렉션 관계가 여러 단계로 중첩되면, Fetch Join은 단계마다 row가 곱연산으로 불어나는 문제가 있었습니다. "@BatchSize"는 이런 다단계 구조에서 각 단계를 독립적인 쿼리로 분리할 수 있어 곱연산 문제 자체를 피할 수 있습니다. 다만 단계가 늘어날수록 쿼리 횟수도 늘어나기 때문에, 각 단계마다 어떻게 적용하고 어디까지 허용할지에 대한 전략이 필요합니다.

---

## 다단계 구조에서의 기본 동작

### 엔티티 구조

```kotlin
@Entity
class Bag(
    @Id @GeneratedValue
    val id: Long? = null,

    @OneToMany(mappedBy = "bag", fetch = FetchType.LAZY)
    @BatchSize(size = 100)
    val items: MutableList<Item> = mutableListOf()
)

@Entity
class Item(
    @Id @GeneratedValue
    val id: Long? = null,

    @ManyToOne(fetch = FetchType.LAZY)
    val bag: Bag? = null,

    @OneToMany(mappedBy = "item", fetch = FetchType.LAZY)
    @BatchSize(size = 100)
    val itemDetails: MutableList<ItemDetail> = mutableListOf()
)
```

"Item"과 "ItemDetail" 사이의 컬렉션에도 동일하게 "@BatchSize"를 적용합니다.

### 쿼리 발생 순서

"Bag" 10개를 조회한 뒤, 각 "Bag"의 "items"에 접근하고, 다시 각 "Item"의 "itemDetails"에 접근하는 시나리오를 보면 다음과 같이 동작합니다.

```sql
-- 1단계: Bag 조회
SELECT * FROM bag;

-- 2단계: items 접근 시, 10개 Bag의 ID를 IN으로 묶어서 한 번에
SELECT * FROM item WHERE bag_id IN (?, ?, ?, ?, ?, ?, ?, ?, ?, ?);

-- 3단계: itemDetails 접근 시, 조회된 모든 Item의 ID를 IN으로 묶어서 한 번에
SELECT * FROM item_detail WHERE item_id IN (?, ?, ?, ..., ?);
```

"Item"이 총 100개라면 3단계 쿼리는 "size" 설정에 따라 1번(size ≥ 100) 또는 여러 번으로 나뉩니다. 핵심은 단계가 늘어나도 쿼리 횟수가 "단계 수"만큼만 선형으로 증가한다는 점입니다. Fetch Join처럼 row가 곱연산으로 불어나는 현상은 발생하지 않습니다.

---

## Fetch Join 다단계 체이닝과의 비교

### Fetch Join의 곱연산 문제 재확인

"Bag" 1개, "Item" 10개, 각 "Item"마다 "ItemDetail" 5개라면 Fetch Join으로 모두 묶을 경우 최종 row 수는 1 × 10 × 5 = 50개입니다. "Bag"과 "Item"의 컬럼 데이터가 각각 50번, 10번 반복 전송되며, 단계가 하나 더 늘어날 때마다 이 곱셈은 기하급수적으로 커집니다.

### @BatchSize의 선형 증가

같은 데이터를 "@BatchSize"로 처리하면 쿼리는 3번(Bag 1번 + Item 1번 + ItemDetail 1번)이고, 각 쿼리의 결과 row 수는 정확히 해당 테이블의 실제 행 수(1, 10, 50)와 일치합니다. 중복 전송이 전혀 없고, 단계가 늘어나도 쿼리 횟수만 선형으로 증가할 뿐 데이터량 폭증은 없습니다.

---

## 각 단계마다 적용 시 고려할 점

### size 값을 단계별로 다르게 줄 필요성

상위 단계("Bag")의 개수가 적고 하위 단계("Item", "ItemDetail")로 갈수록 개수가 많아지는 구조가 일반적입니다. 따라서 하위 단계일수록 "size"를 더 크게 설정하는 것이 유리할 수 있습니다. 다만 "size"를 지나치게 크게 잡으면 IN 절 파라미터 수가 늘어나면서 쿼리 파싱이나 실행 계획 수립 비용이 커질 수 있어, 일반적으로 100~1000 사이에서 데이터베이스 종류와 실제 데이터 분포를 보며 조정합니다.

### 전역 설정 vs 개별 어노테이션

"hibernate.default_batch_fetch_size"로 전역 설정을 두면 모든 컬렉션에 일괄 적용되어 관리가 편하지만, 컬렉션마다 크기 분포가 다르다면 특정 연관관계에만 "@BatchSize(size = N)"을 개별로 덮어써서 세밀하게 조정하는 것이 더 정확합니다.

### 쿼리 횟수가 누적되는 한계

단계가 3~4단계를 넘어가면 "@BatchSize"를 적용해도 결국 단계 수만큼 쿼리가 발생하는 것은 피할 수 없습니다. 곱연산보다는 훨씬 낫지만, 단계가 매우 깊은 구조라면 쿼리 횟수 자체가 누적되어 라운드트립 비용이 커질 수 있습니다. 이런 경우는 애초에 엔티티 그래프를 깊게 가져오는 대신, 필요한 단계까지만 끊어서 DTO 프로젝션이나 별도 조회로 전환하는 설계를 고려하는 것이 일반적입니다.

---

## 실무에서의 일반적인 접근

### 깊이가 2단계 이하일 때

"Bag → Item"처럼 2단계 정도는 "@BatchSize"나 Fetch Join 모두 무리 없이 사용할 수 있습니다. 컬렉션 크기와 페이징 필요 여부에 따라 선택하면 됩니다.

### 깊이가 3단계 이상일 때

"Bag → Item → ItemDetail"처럼 3단계 이상으로 깊어지면, 화면에 실제로 필요한 데이터가 무엇인지 다시 점검하는 것이 우선입니다. 대부분의 경우 최하위 단계까지 엔티티 그래프 전체를 즉시 로딩할 필요는 없고, 화면이나 API 응답에 필요한 필드만 모아서 DTO로 직접 조회(QueryDSL, JPQL의 "new" 연산자 등)하는 방식이 쿼리 횟수와 데이터 전송량 모두에서 더 효율적인 경우가 많습니다.

---

## 전체 요약

다단계 연관관계에서 "@BatchSize"는 각 단계를 독립적인 쿼리로 분리해서 처리하기 때문에, Fetch Join 체이닝에서 발생하는 row의 곱연산 폭증을 피할 수 있습니다. 쿼리 횟수는 단계 수만큼 선형으로 증가하지만 각 쿼리의 row 수는 실제 데이터 수와 정확히 일치해 중복 전송이 없습니다. 각 연관관계마다 데이터 분포에 맞게 "size"를 다르게 설정하는 것이 유리하며, 깊이가 3단계 이상으로 깊어지는 경우에는 "@BatchSize"로 단계를 줄이는 것보다 필요한 필드만 DTO로 직접 조회하는 설계가 더 근본적인 해결책이 될 수 있습니다.

---

# default_batch_fetch_size 전역 설정의 부작용

## 개요

"hibernate.default_batch_fetch_size"를 전역으로 설정하면 애플리케이션 내 모든 지연 로딩 컬렉션과 "@ManyToOne"/"@OneToOne" 연관관계에 일괄적으로 배치 조회가 적용됩니다. 개별 컬렉션마다 "@BatchSize"를 붙이는 수고를 덜어주지만, "모든 곳에 동일한 값이 적용된다"는 특성 때문에 데이터 분포가 제각각인 실제 서비스에서는 여러 부작용을 낳을 수 있습니다.

---

## 컬렉션 크기 편차에 따른 비효율

### size보다 작은 컬렉션이 많을 때

전역 "size"를 100으로 설정했는데 대부분의 부모 엔티티가 자식을 2~3개씩만 가지고 있다면, IN 절에 불필요하게 많은 파라미터 슬롯이 할당되는 것은 아니지만, 한 번의 배치로 가져오는 의미가 크지 않습니다. 오히려 작은 컬렉션이 대부분인 상황에서는 더 작은 "size"(예: 10~20)로도 충분히 N+1을 해소할 수 있는데, 불필요하게 큰 값을 전역으로 깔아두면 IN 절 파싱 비용만 늘어날 수 있습니다.

### size보다 훨씬 큰 컬렉션이 섞여 있을 때

반대로 특정 연관관계만 자식이 수천 개씩 달리는 경우, 전역 "size"가 100이라면 그 컬렉션은 수십 번의 IN 쿼리로 쪼개져서 조회됩니다. 이런 핫스팟 연관관계가 있다면 전역 설정만으로는 부족하고, 해당 연관관계에 개별 "@BatchSize"로 더 큰 값을 따로 지정해야 합니다.

---

## 의도하지 않은 연관관계까지 배치 조회 적용

### ManyToOne/OneToOne에도 적용됨

"default_batch_fetch_size"는 컬렉션뿐 아니라 "@ManyToOne", "@OneToOne"의 지연 로딩 프록시에도 적용됩니다. 예를 들어 "Item.bag"이 LAZY로 설정되어 있다면, 여러 "Item"을 조회한 뒤 각각의 "bag"에 접근할 때도 "bag_id IN (...)" 형태의 배치 조회가 자동으로 일어납니다.

#### 단일 연관관계라면 큰 의미가 없는 경우

이미 "Item" 자체가 "bag_id" 외래키 컬럼을 가지고 있어서, "Bag"을 조회하지 않고 "item.bag.id"만 필요한 상황이라면 프록시 상태로도 충분히 처리 가능한데, 코드 어딘가에서 "item.bag.name"처럼 실제 속성에 접근하는 순간 의도치 않게 배치 쿼리가 추가로 발생할 수 있습니다. 전역 설정 때문에 "어디서 추가 쿼리가 발생하는지"를 추적하기 어려워지는 경우가 생깁니다.

---

## 디버깅과 추적의 어려움

### 쿼리 로그에서 출처 파악이 힘들어짐

전역 설정으로 모든 연관관계에 배치 조회가 적용되면, 쿼리 로그에 "IN (...)" 형태의 쿼리가 애플리케이션 전반에서 빈번하게 나타나게 됩니다. 특정 화면이나 API에서 예상치 못한 쿼리가 발생했을 때, 그것이 의도한 배치 조회인지 아니면 설계상 누락된 연관관계 접근인지 구분하기가 어려워질 수 있습니다.

### 명시적 @BatchSize 대비 가시성 저하

엔티티 코드에 "@BatchSize(size = 100)"이 명시되어 있으면 그 연관관계가 배치 조회 대상이라는 것을 코드만 보고 알 수 있습니다. 반면 전역 설정에만 의존하면 엔티티 코드에는 아무 흔적이 남지 않아, 코드 리뷰나 신규 인원 온보딩 시 "이 연관관계가 배치로 처리되는지"를 설정 파일까지 확인해야 알 수 있습니다.

---

## 메모리와 영속성 컨텍스트 부담

### 한 트랜잭션에서 여러 컬렉션이 동시에 배치 로딩될 때

여러 종류의 컬렉션을 같은 트랜잭션에서 차례로 순회하면, 각 컬렉션마다 전역 "size"만큼 데이터를 한꺼번에 영속성 컨텍스트에 적재합니다. 컬렉션 종류가 많은 화면(예: 대시보드성 API)에서는 의도치 않게 여러 배치 쿼리가 동시다발적으로 발생하면서 영속성 컨텍스트에 올라가는 엔티티 수가 누적되어 메모리 사용량이 예상보다 커질 수 있습니다.

---

## 실무에서의 권장 접근

### 작은 전역 값 + 핫스팟에 개별 오버라이드

일반적으로는 전역 값을 보수적으로 작게(10~50 정도) 설정해 대부분의 N+1을 기본적으로 완화하고, 컬렉션 크기가 유독 크거나 자주 조회되는 특정 연관관계에는 "@BatchSize"를 개별로 덮어써서 더 큰 값을 지정하는 방식이 많이 쓰입니다.

### 모니터링 도구와 함께 사용

Hibernate "Statistics"나 p6spy 같은 쿼리 로깅 도구를 함께 사용해서, 전역 설정 적용 후 실제로 어떤 연관관계에서 몇 번의 배치 쿼리가 발생하는지 주기적으로 확인하는 것이 부작용을 조기에 발견하는 데 도움이 됩니다.

---

## 전체 요약

"default_batch_fetch_size"를 전역으로 설정하면 모든 지연 로딩 컬렉션과 "ManyToOne"/"OneToOne" 연관관계에 일괄적으로 배치 조회가 적용되어 관리는 편해지지만, 컬렉션 크기 편차가 큰 실제 서비스에서는 작은 컬렉션엔 불필요하게 크고 큰 컬렉션엔 부족한 "size"가 적용되는 비효율이 생길 수 있습니다. 또한 의도하지 않은 단일 연관관계까지 배치 조회 대상이 되면서 어디서 추가 쿼리가 발생하는지 추적하기 어려워지고, 여러 컬렉션이 한 트랜잭션에서 동시에 배치 로딩되면 영속성 컨텍스트의 메모리 부담도 커질 수 있습니다. 실무에서는 보수적인 전역 값을 깔아두고 핫스팟 연관관계에만 개별 "@BatchSize"로 오버라이드하는 방식이 일반적으로 권장됩니다.

# Fetch Join과 @BatchSize 선택 기준에 대한 검토

## 개요

"단순/소량/단건성"과 "복잡/대량/페이징"을 가르는 기준으로 Fetch Join과 LAZY+BatchSize를 나누는 판단 기준에 대한 검토입니다. 이는 실무에서 가장 널리 쓰이는 의사결정 방식과 거의 정확히 일치합니다. 몇 가지 세부 조건을 덧붙이면 더 견고한 기준이 됩니다.

---

## 제시된 기준에 대한 평가

### Fetch Join이 적합한 경우

연관관계가 적고(보통 1~2개), 데이터 양이 적고, 페이징이 없는 단건 또는 소량 조회라면 Fetch Join으로 한 번에 가져오는 것이 맞습니다. 쿼리 1번으로 끝나고 row 중복으로 인한 부담도 미미하기 때문에, 코드의 단순함과 성능 모두에서 이득입니다.

### LAZY + @BatchSize가 적합한 경우

연관관계가 많거나 중첩이 깊고, 데이터가 많거나 페이징이 필요한 경우 LAZY로 두고 "@BatchSize"로 N+1을 완화하는 방향도 정확합니다. 페이징과의 호환성, row 중복 방지라는 핵심 이유가 정확히 부합합니다.

---

## 추가로 고려하면 좋을 세부 조건

### "연관관계가 적다"의 기준을 컬렉션 개수로 한정

ToOne 관계("@ManyToOne", "@OneToOne")는 컬렉션이 아니라서 Fetch Join을 여러 개 걸어도 row 폭증 문제가 없습니다. 따라서 "연관관계가 적다"는 기준은 정확히는 "Fetch Join 대상 컬렉션("@OneToMany", "@ManyToMany")이 1개 이하"로 좁히는 것이 더 정확합니다. ToOne 관계는 개수와 무관하게 Fetch Join에 자유롭게 추가해도 괜찮은 경우가 많습니다.

```kotlin
@Query("""
    SELECT i FROM Item i
    JOIN FETCH i.bag
    JOIN FETCH i.category
    WHERE i.id = :id
""")
fun findItemWithBagAndCategory(@Param("id") id: Long): Item?
```

이런 식으로 ToOne 관계 여러 개를 동시에 Fetch Join해도 row 수는 늘어나지 않으므로, 컬렉션이 끼지 않는다면 "연관관계 개수"는 크게 신경 쓰지 않아도 됩니다.

### 컬렉션이 정확히 1개일 때는 페이징 없어도 Fetch Join + DISTINCT로 충분

컬렉션이 1개뿐이라면 페이징이 없는 경우 Fetch Join만으로 충분하고, "DISTINCT"도 보통 불필요합니다(영속성 컨텍스트 동일성 보장 덕분). 컬렉션이 2개 이상으로 늘어나는 순간부터 "MultipleBagFetchException"이나 카티전 곱 문제가 생기므로, "컬렉션 1개 vs 2개 이상"이 실질적인 분기점이 됩니다.

### "데이터가 많다"의 기준은 컬렉션 크기뿐 아니라 부모 컬럼 무게도 함께

부모 엔티티에 큰 텍스트 컬럼이나 BLOB성 데이터가 있다면, 자식 컬렉션 크기가 작아도(예: 5~10개) row 중복으로 인한 전송량이 의외로 커질 수 있습니다. 단순히 "자식이 몇 개냐"만 보지 말고 "부모 row 자체가 얼마나 무거운가"도 함께 고려하는 것이 좋습니다.

---

## 정리된 의사결정 기준

| 조건 | Fetch Join | LAZY + @BatchSize |
| --- | --- | --- |
| 컬렉션(ToMany) 개수 | 0~1개 | 2개 이상 |
| 페이징 | 없음 | 있음 |
| 데이터 규모 | 소량 | 대량 |
| 연관관계 중첩 깊이 | 1단계 | 2단계 이상 |
| 부모 row 무게 | 가벼움 | 무거움(중복 전송 부담) |

ToOne 관계는 이 표와 무관하게 항상 Fetch Join에 자유롭게 추가 가능합니다.

---

## "적당히 찢는다"는 표현에 대한 보완

"batchsize 이용해 하위 테이블을 in절로 불러와서 n+1 쿼리증가를 너무 증가시키지 않게 적당히"라는 부분에서, 다단계 구조라면 단계 수만큼 쿼리가 선형으로 늘어나는 것은 구조적으로 피할 수 없는 부분입니다. 여기서 "적당히"를 결정하는 실질적인 레버는 "size" 값 자체보다는 "몇 단계까지 엔티티 그래프로 따라가느냐"입니다. 2단계까지는 "@BatchSize"로 충분히 감당되지만, 3단계 이상으로 깊어진다면 그 시점부터는 DTO 직접 조회로 전환하는 것이 "쿼리 횟수를 적당히 유지하는" 더 근본적인 방법이라는 점을 기준에 추가하면 좋습니다.

---

## 전체 요약

"단순/소량/비페이징은 Fetch Join, 복잡/대량/페이징은 LAZY+BatchSize"라는 기준은 실무에서 널리 쓰이는 기준과 정확히 일치합니다. 여기에 ToOne 관계는 컬렉션과 무관하게 항상 Fetch Join에 자유롭게 추가 가능하다는 점, 컬렉션이 정확히 1개일 때는 페이징만 없으면 DISTINCT 없이도 Fetch Join으로 충분하다는 점, 부모 엔티티의 컬럼 무게도 데이터 규모 판단에 포함시켜야 한다는 점을 보완하면 더 견고한 기준이 됩니다. 다단계 중첩에서는 "@BatchSize"의 "size" 조정보다 "몇 단계까지 엔티티 그래프로 따라갈지"를 제한하는 것이 더 본질적인 해결책입니다.
