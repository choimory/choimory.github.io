# HTTP/1.1과 HTTP/2의 차이

- HTTP/1.1과 HTTP/2는 모두 **웹 상에서 클라이언트(브라우저)와 서버 간의 데이터 통신을 위한 프로토콜**이지만, HTTP/2는 HTTP/1.1의 **성능 문제**를 해결하고 **효율성을 높이기 위해** 개발된 개선된 버전이다.
- HTTP/2는 페이지 로딩 속도, 연결 효율성, 데이터 전송 최적화를 통해 웹 성능을 크게 향상시켰다. 주요 차이점은 다음과 같다.

---

# **HTTP/1.1**

- HTTP/1.1은 웹 통신을 위한 가장 오래된 프로토콜 중 하나로, 1997년에 도입되어 오랜 기간 동안 표준으로 자리 잡아왔다.
- 그러나 여러 가지 성능 제약과 한계가 존재한다.

## 특징

### **텍스트 기반**:

- 요청과 응답은 텍스트 형태로 이루어진다.
- 클라이언트와 서버 간의 데이터는 사람이 읽을 수 있는 텍스트 형식으로 전송된다.

### **단일 요청/응답**:

- HTTP/1.1은 한 번의 요청에 하나의 응답만 처리할 수 있다.
- 이를 **단방향 통신**이라고 하며, 클라이언트가 서버에 요청하면 그에 대한 응답을 받을 때까지 대기해야 한다.

### **지속 연결**:

- 기본적으로 요청/응답 후 연결이 끊기지만, `Keep-Alive` 헤더를 통해 연결을 지속적으로 유지할 수 있다.
- 그러나 많은 연결을 유지하는 것은 **과부하**를 초래할 수 있다.

### **헤더 오버헤드**:

- HTTP/1.1에서는 매 요청마다 헤더가 반복적으로 전송된다.
- 동일한 리소스에 대한 여러 요청이 있을 때도 동일한 헤더 정보가 여러 번 보내지므로 **네트워크 대역폭**을 낭비한다.

### **HOL(Head-of-Line) 차단 문제**:

- 요청을 순차적으로 처리해야 하기 때문에, 하나의 요청이 지연되면 나머지 요청들도 처리되지 못하는 문제가 발생한다.

## 한계

### **성능 저하**:

- 다수의 요청이 순차적으로 처리되어 성능이 저하될 수 있으며, 특히 현대 웹 페이지에서 이미지, CSS, JavaScript 등 많은 리소스를 한 번에 불러와야 할 때 속도가 느려진다.

### **한정된 멀티플렉싱**:

- 여러 리소스를 동시에 요청하는 경우, 여러 TCP 연결을 생성하거나 요청을 순차적으로 보내야 하므로 효율적이지 않다.

---

# **HTTP/2**

- HTTP/2는 HTTP/1.1의 성능 문제를 해결하기 위해 2015년에 도입된 프로토콜로, **속도**와 **효율성**에 중점을 두고 설계되었다.

## 특징

### **이진 전송**:

- HTTP/2는 요청과 응답을 **이진(binary)** 형태로 전송한다.
- 이를 통해 텍스트 기반보다 훨씬 더 빠르고 효율적인 데이터 처리가 가능하다.

### **멀티플렉싱**:

- HTTP/2에서는 하나의 연결을 통해 **다수의 요청과 응답을 동시에 처리**할 수 있다.
- 이로 인해 한 요청이 지연되더라도 다른 요청이 대기하지 않고 병렬로 처리될 수 있어, **HOL 차단 문제**가 해결된다.

### **헤더 압축**:

- **HPACK**이라는 헤더 압축 방식을 도입하여, 중복된 헤더를 압축하고 캐시로 관리한다.
- 이를 통해 네트워크 **대역폭을 절약**할 수 있다.

### **서버 푸시**:

- HTTP/2는 서버가 클라이언트가 요청하지 않은 리소스를 미리 전송하는 **서버 푸시** 기능을 지원한다.
- 예를 들어, HTML 파일을 요청할 때 서버는 클라이언트가 필요할 CSS, JavaScript 파일을 미리 전송할 수 있다.

### **단일 연결**:

- HTTP/2는 한 번의 TCP 연결로 여러 요청을 처리하기 때문에, 연결 설정 비용을 줄이고 서버에 과부하를 줄여준다.

### **우선순위**:

- 요청 간의 **우선순위를 지정**할 수 있어, 중요한 리소스(예: CSS, JavaScript 등)를 먼저 전송하여 페이지 로딩 시간을 단축시킬 수 있다.

## 장점

### **성능 향상**:

- 멀티플렉싱, 헤더 압축, 서버 푸시 등의 기술 덕분에 HTTP/1.1보다 훨씬 빠르고 효율적으로 웹 리소스를 로딩할 수 있다.

### **더 나은 네트워크 효율성**:

- 단일 연결을 통해 여러 요청을 처리하고, 중복된 헤더 전송을 줄여 네트워크 사용량을 크게 절감할 수 있다.

---

# 비교

| **특징** | **HTTP/1.1** | **HTTP/2** |
| --- | --- | --- |
| **전송 형식** | 텍스트 기반 | 이진(binary) 기반 |
| **멀티플렉싱** | 불가능 (순차 처리) | 가능 (동시 처리) |
| **헤더 압축** | 없음 | HPACK 방식으로 압축 |
| **서버 푸시** | 지원하지 않음 | 지원 |
| **연결 수** | 다중 연결 필요 | 단일 연결로 처리 |
| **HOL 차단** | 문제 발생 | 해결됨 |
| **우선순위** | 없음 | 요청 간 우선순위 설정 가능 |
| **사용된 주요 버전** | 오랜 기간 표준으로 사용 | 최신 웹 애플리케이션에서 널리 사용됨 |

---

# 결론

- HTTP/1.1은 오래된 표준이지만 여전히 많은 웹에서 사용되고 있다.
- 그러나 HTTP/2는 멀티플렉싱, 헤더 압축, 서버 푸시와 같은 기능을 통해 **더 빠르고 효율적인 통신**을 제공하여, 특히 **현대의 복잡한 웹 애플리케이션**에서 성능을 크게 향상시킨다.
- HTTP/2는 기존 웹의 한계를 극복하기 위한 진화된 프로토콜이다.
