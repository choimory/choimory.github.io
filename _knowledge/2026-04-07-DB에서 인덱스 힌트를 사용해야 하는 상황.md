# DB에서 인덱스 힌트를 사용해야 하는 상황

- *인덱스 힌트(Index Hint)*는 데이터베이스가 쿼리를 최적화할 때 사용할 인덱스를 **명시적으로 지정**하는 방법이다.
- 일반적으로 데이터베이스 최적화 엔진은 쿼리에 적합한 인덱스를 자동으로 선택하지만, **특정 상황에서 인덱스 힌트를 사용하는 것이 유리**할 수 있다.

---

# **잘못된 인덱스 선택으로 인한 성능 저하**

- 데이터베이스의 최적화 엔진이 잘못된 인덱스를 선택하여 **비효율적인 실행 계획**을 사용하게 되는 경우가 있다.
- 이때, 쿼리 성능이 저하될 수 있으며, 의도한 대로 최적의 인덱스를 선택하지 못할 수 있다.
- **예시**:
    - 복수의 인덱스가 존재하는 테이블에서, 데이터베이스 엔진이 상대적으로 **비효율적인 인덱스**를 선택하는 경우, 인덱스 힌트를 사용해 더 적합한 인덱스를 강제로 사용하게 할 수 있다.
    
    ```sql
    SELECT * FROM orders USE INDEX(order_date_idx) WHERE order_date > '2023-01-01';
    ```
    

---

# **복잡한 쿼리에서 특정 인덱스 강제 적용**

- 복잡한 쿼리에서 데이터베이스가 자동으로 선택하는 인덱스보다 더 나은 성능을 제공할 인덱스를 **수동으로 지정**해야 할 때 인덱스 힌트를 사용한다.
- **예시**:
    - 조인, 서브쿼리 등 복잡한 쿼리에서 데이터베이스 엔진이 잘못된 인덱스를 사용하면 성능 저하가 발생할 수 있다. 이때, 인덱스 힌트를 통해 **명시적으로 올바른 인덱스를 강제**할 수 있다.
    
    ```sql
    SELECT c.customer_id, o.order_id
    FROM customers c USE INDEX(customer_name_idx)
    JOIN orders o ON c.customer_id = o.customer_id
    WHERE c.customer_name = 'John Doe';
    ```
    

---

# **통계 정보가 부정확한 경우**

- 데이터베이스는 인덱스를 선택할 때 **통계 정보**를 기반으로 쿼리 계획을 세운다.
- 하지만, 통계 정보가 오래되었거나 부정확한 경우, 잘못된 인덱스를 선택할 수 있다. 이러한 경우에 인덱스 힌트를 사용해 **올바른 인덱스를 강제**할 수 있다.
- **예시**:
    - 테이블에 저장된 데이터가 크게 변화하여 통계 정보가 오래되었지만, 통계를 다시 수집할 시간이 없을 때, 인덱스 힌트를 통해 성능 저하를 방지할 수 있다.
    
    ```sql
    SELECT * FROM products USE INDEX(product_category_idx) WHERE category = 'Electronics';
    ```
    

---

# **다중 인덱스가 존재하는 상황에서 최적의 인덱스 선택**

- 테이블에 다수의 인덱스가 존재할 때, 어떤 인덱스가 가장 효율적인지 **최적화 엔진이 정확하게 판단하지 못할 수 있다**.
- 이때, 인덱스 힌트를 사용해 성능이 좋은 인덱스를 선택하도록 강제할 수 있다.
- **예시**:
    - 여러 인덱스가 있는 테이블에서 특정 인덱스가 가장 효율적일 때, 이를 강제 사용하기 위해 인덱스 힌트를 적용할 수 있다.
    
    ```sql
    SELECT * FROM employees USE INDEX(emp_department_idx) WHERE department_id = 10;
    ```
    

---

# **전체 테이블 스캔을 피하고 싶을 때**

- 데이터베이스가 인덱스를 무시하고 **전체 테이블 스캔(Full Table Scan)**을 수행하는 경우, 인덱스 힌트를 사용해 **인덱스 검색을 강제**할 수 있다.
- 테이블에 데이터가 많을 경우, 전체 테이블 스캔은 매우 느릴 수 있다.
- **예시**:
    - 테이블이 커지고 데이터가 많은 경우, 인덱스를 사용하지 않으면 성능이 크게 저하된다.
    - 이때 인덱스 힌트를 사용해 인덱스를 강제하면 성능을 개선할 수 있다.
    
    ```sql
    SELECT * FROM sales FORCE INDEX(sales_date_idx) WHERE sales_date = '2023-09-01';
    ```
    

---

# **데이터 분포가 비대칭적일 때**

- 테이블에 저장된 데이터의 **분포가 불균형**할 경우, 데이터베이스가 전체 테이블 스캔을 선택할 가능성이 높다.
- 이때 인덱스 힌트를 통해 **인덱스를 강제 적용**하여 쿼리 성능을 높일 수 있다.
- **예시**:
    - 특정 조건의 데이터가 테이블에서 매우 많이 존재하거나, 매우 적게 존재하는 경우, 인덱스 사용이 더 유리할 수 있다.
    
    ```sql
    SELECT * FROM orders FORCE INDEX(order_status_idx) WHERE order_status = 'Pending';
    ```
    

---

# **테이블의 크기가 매우 큰 경우**

- **대규모 테이블**에서 특정 조건의 데이터를 조회할 때, 전체 테이블 스캔이 발생하면 성능이 크게 저하된다.
- 이때 인덱스 힌트를 사용하여 특정 인덱스를 강제하면 성능이 개선될 수 있다.
- **예시**:
    - 수백만 개의 레코드를 가진 테이블에서 특정 인덱스를 사용하여 원하는 데이터를 빠르게 조회하고자 할 때 인덱스 힌트를 사용한다.
    
    ```sql
    SELECT * FROM large_table USE INDEX(large_table_idx) WHERE column_x = 'value';
    ```
    

---

# **특정 인덱스가 최신 데이터에 적합할 때**

- 데이터가 자주 변경되는 테이블에서 **특정 인덱스가 최신 데이터에 더 적합**할 수 있다.
- 이런 경우, 최적화 엔진이 선택하는 기본 인덱스보다 최신 데이터를 잘 처리할 수 있는 인덱스를 명시적으로 지정해야 할 수 있다.
- **예시**:
    - 인덱스 힌트를 사용하여 **최신 데이터**에 최적화된 인덱스를 선택할 수 있다.
    
    ```sql
    SELECT * FROM logs FORCE INDEX(new_data_idx) WHERE log_date > '2024-01-01';
    ```
    

---

# **비용 기반 최적화가 잘못된 판단을 할 때**

- **비용 기반 최적화(CBO, Cost-Based Optimizer)**는 비용 추정에 기반해 실행 계획을 세운다.
- 하지만, 비용이 잘못 추정되거나, 통계 정보가 부족할 때 비효율적인 인덱스를 선택할 수 있다.
- 이때 인덱스 힌트를 사용해 데이터베이스 엔진이 더 효율적인 인덱스를 사용하게 할 수 있다.
- **예시**:
    - 최적화 엔진이 잘못된 비용을 기반으로 비효율적인 쿼리 실행 계획을 세울 때, 인덱스 힌트를 통해 더 적합한 인덱스를 선택할 수 있다.
    
    ```sql
    SELECT * FROM transactions USE INDEX(tx_date_idx) WHERE transaction_date BETWEEN '2023-01-01' AND '2023-12-31';
    ```
    

---

# 정리

- 인덱스 힌트는 데이터베이스 최적화 엔진이 **비효율적인 실행 계획**을 세우거나 **잘못된 인덱스**를 선택할 때 성능을 개선하는 데 유용하다.
- 주로 **복잡한 쿼리**, **데이터 분포 불균형**, **대규모 테이블**에서 사용하며, 특정 쿼리에 대해 명시적으로 **최적의 인덱스**를 선택하여 성능을 최적화할 수 있다.
- 다만, 과도하게 인덱스 힌트를 사용하면 **유지보수의 어려움**이 발생할 수 있으므로, **필요한 상황에서만 신중하게 적용**하는 것이 중요하다.
